# Финансовый контур Yakkasaroy — инварианты и процедуры

Цель: не нарушить принципы ХМС/ManaJet и инварианты БД при работе с деньгами. Эти
правила держатся триггерами и констрейнтами Postgres, а не клиентом. Нарушение даёт
расхождение балансов или порчу аудита, которые трудно откатить.

Перед любой задачей про деньги/БД свериться с актуальным состоянием схемы: последний по
времени файл в `supabase/migrations/` и статусы в `supabase/README.md`. Строить код и
изменения поверх свежего состояния, а не baseline.

## Семь железных правил (не нарушать)

1. **Деньги — только через Реестр `fp_register`.** Балансы фондов (`funds.balance`) и
   счетов ДС (`cash_accounts.balance`) — производные леджера, их поддерживают триггеры.
   **Никогда не писать `balance` напрямую** (ни в SQL, ни в API). Чтобы изменить деньги —
   вставить операцию в Реестр через серверную RPC.
2. **Реестр неизменяем.** Триггер `fp_register_no_update` запрещает `UPDATE` строк
   Реестра. Откат операции — только **компенсирующей записью** (см. `fp_reverse_fund_op`),
   а не правкой/удалением. RLS-политики `UPDATE` на `fp_register` нет — не добавлять.
3. **Запрет минуса.** Триггер не даёт балансу уйти в минус — операция отклоняется на
   уровне БД. Не обходить проверкой на клиенте.
4. **Закрытый период заблокирован.** Операции в закрытом периоде ФП запрещены триггером.
   Период закрывается Директивой (`fp_close_period`), переоткрывается `fp_reopen_period`
   (оба пишут в `audit_log`).
5. **Только фактически имеющиеся средства.** Нельзя одобрить заявку/счёт на сумму больше
   «Доступно» в фонде — держат триггеры `trg_request_approve_funds_check` /
   `trg_bill_approve_funds_check` при переводе в `approved`.
6. **Каждая запись привязана к точке** (`location_id`). Точка/филиал — первоклассная
   сущность; владелец видит консолидированно через RLS.
7. **Справочники-коды реальны.** Фонды ФД1–ФД9/1, статьи РД1–РД9, D-коды доходов,
   7 отделений, состояния и коэффициенты ЗП — стартовые данные системы. Не переименовывать,
   не «чистить», не выдумывать новые в коде.

## Как двигать деньги: только через RPC

Никогда не вставлять в `fp_register` напрямую и не трогать `balance`. Использовать
серверные функции (перемещения/займы пишут **парные** записи через `pair_id`):

| Действие | RPC | Заметки |
|---|---|---|
| Доход → распределение по фондам | вставка в `incomes` (триггер `trg_income_to_register`) | доход сам создаёт запись Реестра; распределение по этапам — `fp_distribute_stage` |
| Перемещение фонд→фонд | `fp_fund_transfer` | проверяет `no_transfer` |
| Заём фонд→фонд / возврат | `fp_fund_loan` / `fp_fund_loan_return` | сальдо в колонке «Долг» |
| Ручной приход / возврат (изъятие) фонда | `fp_fund_income` / `fp_fund_return` | проверяют `no_transfer`; накопительный фонд блокирует |
| Оплата заявки ЗРС | `fp_pay_request` | блокирует строку `for update` (анти-двойная оплата); платит `coalesce(approved_amount, planned_amount)`; запрет накопительного фонда |
| Оплата счёта поставщика / счёта клиента / ЗП | `fp_pay_bill` / `fp_pay_invoice` / `fp_pay_payroll` | |
| Перевод между счетами ДС | `fp_cash_transfer` | |
| Откат операции фонда | `fp_reverse_fund_op` | компенсирующая запись (не удаление); только в открытом периоде; для `fund_loan` — полный возврат |
| Этапы Директивы | `fp_distribute_stage` / `fp_close_period` / `fp_reopen_period` / `fp_reset_distribution` | период чт–ср |
| Этап фонда | `fp_set_fund_stage` | переносит дефолтное правило `distribution_rules` |
| Сверка балансов | `fp_reconcile_balances` | `SUM(fp_register)` = баланс фонда/счёта |

Точные имена RPC и значения enum `register_op_type` — в `references/finance-domain.md`.
Если нужной операции нет — добавлять **новую RPC + миграцию**, а не писать баланс с клиента.

## Расчёт колонок фонда (вкладка «Фонды»)

- `funds.balance` = **физические деньги** фонда (ведёт триггер Реестра).
- **Остаток** = Σ одобренных, но **не оплаченных** обязательств из фонда
  (`payment_requests`/`supplier_bills`/обязательства). Это **агрегирующий запрос**,
  леджер не меняем.
- **Доступно** = `balance − Остаток`.
- **Долг** = сальдо займов (`fund_loan` − `fund_loan_return`, со знаком: − фонду должны,
  + фонд должен).
- Инвариант: **физические деньги = Доступно + Остаток**.

Первоисточник правил вкладки — `docs/funds-spec.md`. Этап у фонда — одно свойство
(`funds.stage`); правило «ФД6 с двух этапов» **отменено** (фонд одноэтапный).

## Изменение схемы БД (обязательный чек-лист)

При любом изменении схемы (таблицы/функции/триггеры/RLS/индексы):

1. **Только новый файл миграции** в `supabase/migrations/` (`ГГГГММДДЧЧММСС_имя.sql`).
   Применённые файлы **не редактировать**. Стиль идемпотентности:
   `if not exists` / `on conflict` / `drop ... if exists`.
2. **Новые таблицы — всегда с RLS** на родных функциях прав: `my_role()`,
   `is_fin_admin()`, `has_location_access()`, `has_fund_access()`, `holds_position()`
   (+ таблицы персональных прав `user_location_access`, `fund_access`,
   `expense_type_access`). Плюс соглашения: `outer_id`, `is_archived`,
   периоды `starts_on`/`ends_on`, валюта `currency_id` (uuid).
3. **pgTAP-тест инварианта** в `supabase/tests/` (запуск `supabase test db` на
   ветке/staging). Структурные/read-only проверки безопасны везде.
4. **Перегенерировать типы**: `supabase gen types` → `src/lib/database.types.ts`
   (руками не править).
5. **Деньги — точечными блокировками** (`FOR UPDATE`), не слепым `SERIALIZABLE`
   (retry-конфликты).
6. **Применение через Supabase MCP** (skill `supabase-migration`). OAuth-ссылка
   одноразовая и быстро протухает — вызывать `authenticate` заново и давать заказчику
   свежую ссылку. После применения — отметить статус в `supabase/README.md`.

Для пошаговой работы с миграциями использовать встроенный skill **`supabase-migration`**.

## Код фронта/API

- Обращения к БД — через типизированный клиент (`supabase` из `lib/supabase.ts`) и
  API-слой `src/lib/api.js`. Новый чистый код — на TypeScript, с unit-тестами (Vitest).
- Компоненты не мутируют данные напрямую сверх локального state; деньги не считают на
  клиенте в обход Реестра.
- ХМС-термины не переводить и не переименовывать (ФП, ФРС, фонд, Директива, ЦКП, ИЦО,
  ЗРС, шляпа, статистика, Реестр, состояния, туйхона, точка).
- Денежные суммы — утилитами `src/utils/format.ts`: `fmt` (точно, до дирамов) и
  `fmtShort` (млн/тыс для обзоров). Валюта по умолчанию — TJS (сомони).
- Мобильная адаптация — часть каждого изменения (`isMobile`): списки сущностей —
  карточками, а не горизонтальным скроллом.
- Перед PR — зелёные `npm run typecheck`, `npm test`, `npm run lint`, `npm run build`.
