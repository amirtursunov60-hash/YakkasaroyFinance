# Definition of Done и мобильная адаптация — Yakkasaroy

Развиваемся эволюционно, без большого рефакторинга: **touch → extract → type → test**
(Boy Scout Rule — оставляй код чище, чем взял). Фичи не останавливаем ради
инфраструктуры — вместо этого держим планку приёмки. Применять к **каждому** изменению.

## Чек-лист приёмки (перед PR)

1. **Новый код — сразу на TypeScript.** Чистую (не-React) логику выносить в `.ts`-модули
   и покрывать unit-тестами (Vitest). Обращения к БД — через типизированный клиент
   (`supabase` из `lib/supabase.ts`), типы — `lib/database.types.ts`.
2. **Трогаешь старый `.js/.jsx` — попутно переводи затронутое на `.ts/.tsx`** и дописывай
   тесты на критичную логику. Не обязательно переписывать весь файл — достаточно того,
   что меняешь.
3. **Мобильная адаптация выполнена** (см. ниже) — это часть изменения, не отдельная задача.
4. **Стиль из темы** — никаких захардкоженных цветов (см. ниже).
5. **Деньги — только через Реестр `fp_register`** (если изменение их касается; детали —
   skill `yakkasaroy-finance`). Изменение схемы БД — миграция + RLS + pgTAP + перегенерация
   типов (skill `supabase-migration`).
6. **Зелёные проверки** — обязательны перед PR:
   ```bash
   npm run typecheck   # tsc --noEmit
   npm test            # Vitest один прогон
   npm run lint        # ESLint (ошибки хуков/необъявленных имён блокируют)
   npm run build       # Vite сборка
   ```
7. **~20% ёмкости спринта — на операционный контур** (staging, restore-drill бэкапов,
   плановая сверка/мониторинг), а не только на фичи.

## Мобильная адаптация — обязательна для каждого UI-изменения

Управляющие работают с телефона. Любой новый/правленый экран, таблица, форма, шапка или
модал сразу делается адаптивным под `isMobile` (из `useIsMobile`) — без отдельной команды
заказчика. После UI-правки мысленно (или прогоном) проверить **обе** ветки — десктоп и
`isMobile` — и довести мобильный вид сразу, не дожидаясь «отодвинь/сократи».

Правила по умолчанию:

- **Списки сущностей (фонды, счета и т.п.) — карточки, а не горизонтально-скроллящиеся
  таблицы** (образец — вкладка «Фонды»): каждая сущность = адаптивная карточка
  (`repeat(auto-fill, minmax(~300px, 1fr))`, на телефоне в одну колонку), все ключевые
  цифры и действия видны без горизонтального скролла; действия — кнопками на карточке.
- **Широкие таблицы:** на телефоне ключевые колонки (название/сумма/действие) помещаются
  в ширину экрана, второстепенные — за ними с горизонтальным скроллом; на телефоне своя
  `gridTemplateColumns` (уже колонки, меньше паддинги).
- **Кнопки/тулбары в шапках** переносятся (`flexWrap`); длинные подписи на телефоне
  сокращаются или прячутся за иконку.
- **Модалы** — ширина `min(Xpx, 100%)`; формы на телефоне в одну колонку.
- **Ничего не вылезает за правый край** и не обрезается — проверять правый блок шапки,
  аватар, чипы.

## Стиль и оформление

- **Всё оформление — в `src/theme/`.** Цвета — только из палитры `C` (`THEMES` в
  `theme/theme.js`); обе темы (dark/light) должны работать. Никаких захардкоженных цветов
  в компонентах. Дефолт — тёмная тема в духе Alif Business (зелёные акценты).
- Общие инлайн-стили — в `makeStyles` (`styles.js`), использовать как `st.имя`;
  hover/анимации — в `makeCss` (`css.js`) через классы. Новых CSS-файлов (кроме
  `src/index.css`) и CSS-фреймворков не заводить.
- Графики — собственные SVG в `src/components/charts/`, библиотеки графиков не подключать.
- Денежные суммы — утилитами `src/utils/format.ts`: `fmt` (точно, до дирамов) и
  `fmtShort` (млн/тыс для дашбордов). Валюта по умолчанию — TJS (сомони).
- Компоненты получают всё через `useTheme()`: `{ C, st, theme, setTheme, lang, setLang,
  isMobile, profile }`.

## Язык и термины

- **Язык интерфейса — русский** (надписи, метки, комментарии). i18n заложен (`lang` ru/tj),
  строки пока в коде.
- **ХМС-термины не переводить и не переименовывать** — ни в UI, ни в данных: ФП, ФРС, фонд,
  Директива, ЦКП, ИЦО, ЗРС, шляпа, статистика, квота, Реестр, состояния (Власть, Изобилие,
  Норма, Чрезвычайное положение, Опасность, Несуществование), боевое планирование, туйхона,
  точка. Глоссарий — Приложение А в ТЗ v2.
- **Моки — в `src/data/`** (не зашивать демо-данные в компоненты). Компоненты не мутируют
  данные напрямую сверх локального state.

## Добавить экран (роутинга нет)

Навигация — на state в `src/components/AppShell.jsx` (`activeModule` + `active`). Чтобы
добавить экран: компонент в `src/modules/<модуль>/`, пункт в `src/data/navigation.js`
(`MODULES`/`MODULE_NAV`), ветка рендера в `AppShell.jsx`.

## PR-воркфлоу (деплой)

- **Каждое изменение — отдельным PR в `main`**; напрямую в `main` не пушить. Так изменение
  получает номер, а Vercel-бот создаёт preview-деплой.
- В ответах заказчику ссылаться на изменение **по номеру PR** (например, «PR #36») и
  присылать **прямую ссылку на preview-деплой Vercel** (из комментария Vercel-бота под PR,
  поле «Preview»).
- Прод на Vercel обновляется после слияния PR в `main`; env-переменные
  (`VITE_SUPABASE_URL`, `VITE_SUPABASE_KEY`) задаются в панели Vercel.

## Связанные скиллы

- `yakkasaroy-finance` — инварианты денег/Реестра при финансовых изменениях.
- `supabase-migration` — создание и применение миграций БД.
