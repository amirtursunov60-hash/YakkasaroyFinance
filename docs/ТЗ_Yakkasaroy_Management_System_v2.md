# Техническое задание v2.0
## Веб-приложение управления компанией «Яккасарой» (Yakkasaroy Management System)

**Версия:** 2.0 (полная переработка v1.0)
**Дата:** июнь 2026
**Заказчик:** сеть ресторанов и туйхон «Яккасарой», Душанбе
**Основа версии:** анализ публичного API ManaJet (api.manajet.org/Help, Swagger) + разбор кода прототипа YakkasaroyFinance.jsx (~3300 строк)
**Целевой стек:** React + Supabase (PostgreSQL)

---

## 1. Общие сведения

### 1.1 Цель
Единая система управления сетью «Яккасарой» (туйхоны, рестораны, кафе в Душанбе, Худжанде, Панджакенте) по технологии ХМС, с финансовым планированием как ядром, масштабируемая до 100 точек к 2030 году.

### 1.2 Текущее состояние прототипа (по результатам анализа кода)

Прототип YakkasaroyFinance.jsx — единый React-каркас с 8 модулями и переключением по сайдбару, тёмная/светлая тема в стиле Alif, мобильная адаптация (useIsMobile), RU-интерфейс с заложенным переключателем языка. Все данные — моки в памяти.

**Реализовано в UI (переносится в продукт как готовый дизайн):**
- **ФП → Директива:** выбор финансовой недели (чт–ср), три уровня распределения (Выручка → Маржинальный доход → Скорректированный доход), редактируемые проценты фондов, кнопки «Рассчитать/Одобрить/Сброс» по уровню, панель заявок с группировкой по отделениям оргсхемы, контроль «остаток = старт + распределено − потрачено».
- **ФП → Доходы:** дерево видов дохода с реальными кодами (D1 Душанбе Яккасарой, D1.1 Марказ, D1.2 Fly Garden, D1.3–D1.4 Фемали 1/2, D1.5 Кейтеринг, D1.6 Прямой фонд) и листьями (ВИП зал, ЛЮКС зал, Оши нахор, Оформление, Торт и т.д.), колонки «Было/Стало», тренды.
- **ФП → Расходы:** дерево статей РД1–РД9 с подстатьями (Продукты, Маркетинг, Зарплата, Налоги, Оборудование, Административные).
- **ФП → Фонды:** реальные фонды ФД1–ФД9/1, типы рабочий/накопительный, перемещение и заём фонд→фонд с возвратом, история операций, запрет ухода в минус. **Важно: ФД6 (Резервы) пополняется на двух уровнях распределения одновременно.**
  > ⚠️ ОТМЕНЕНО решением заказчика (см. docs/funds-spec.md §10–12): фонд одноэтапный, этап — свойство фонда (`funds.stage`), смена этапа — RPC `fp_set_fund_stage`.
- **ФП → Заявки:** карточки с кодом РД, постом заявителя (формат «7.20.1 Начальник, ФИО»), суммой, способом оплаты, фондом-источником, статусом.
- **ФП → Контроль средств:** счета ДС (кассы точек, счета Алиф TJS/USD), ввод факта vs расчёт.
- **ФП → Счета поставщиков / клиентов:** счета с привязкой к статье РД и точке; клиентские счета банкетов с предоплатой и долгом.
- **ФП → Расчёт зарплаты:** баллы сотрудников × коэффициент состояния статистики (Власть 1.3 … Опасность 0.7), деление ФОТ по баллам.
- **ФП → Управленческие отчёты:** ДДС, P&L, сравнение точек.
- **Статистики:** ИЦО-доска, графики 12 недель, автоопределение состояния по тренду 4 недель, флаг invert для «обратных» статистик (жалобы, ФОТ%).
- **Оргсхема:** 7 отделений, посты, шляпы (библиотека HAT_LIB), вакансии.
- **CRM:** воронка банкетов (заявка → показ → предложение → договор → выигран), база клиентов, брони залов.
- **Личный кабинет:** мой кабинет, боевое планирование, задачи.
- **Модуль «Ресторан»:** заказы, столы, меню, склад, смены — собственная разработка, в ManaJet аналога нет (см. 5.11).

**Не реализовано (заглушки/моки):** авторизация (декоративный Login), раздел «Обязательства», модуль «Сотрудники», «Управление проектами», сохранение данных, права ролей.

### 1.3 Задача
Превратить прототип в клиент-серверную систему на Supabase, воспроизведя функциональную модель ManaJet (по его реальному API, см. раздел 3) с адаптацией под мультиточечный ресторанный бизнес.

### 2.3 Ключевые принципы модели ManaJet (подтверждены API, обязательны к сохранению)
1. **Операция дохода сразу указывает, куда физически пришли деньги** (счёт ДС + способ оплаты). Контроль средств — производная от этих привязок, а не отдельный ввод.
2. **Заявка подаётся в формате ЗРС** — три обязательных текстовых поля: данные, ситуация, предлагаемое решение — и **от поста оргсхемы**, а не от «пользователя».
3. **Счёт поставщика живёт в двух периодах**: период одобрения и период оплаты могут отличаться (одобрили на этой неделе, оплатили через две).
4. **Реестр (Register) — единая лента операций ФП** с типом операции; всё движение денег видно в одном месте.
5. **Статистики имеют квоты** (план на период) и периоды разных типов.
6. **У всех сущностей есть outer_id** — поле для интеграций (iiko, импорт), архивирование вместо удаления (in_archive), папочная организация справочников.
7. Доход может быть **возвратом** (is_return) — возвраты клиентам уменьшают доход, не являясь расходом.

## 3. Роли и права доступа

| Роль | Доступ |
|---|---|
| Владелец | Всё, все точки, закрытые фонды |
| Финансовый директор | Весь блок ФП, закрытые фонды, проведение Директивы, закрытие периодов |
| Операционный директор / Куратор | Статистики, задачи, Поток, оргсхема, заявки по своим точкам; финансы — просмотр |
| Управляющий точкой | Свои точки: ввод доходов (с указанием счёта ДС и способа оплаты), заявки, статистики, задачи |
| Бухгалтер | Контроль средств, отметка оплат по одобренным заявкам/счетам, ведомости, без права одобрения |
| Сотрудник | ЛК: задачи, статистики, подача заявок (ЗРС), шляпы, сообщения |

Требования (как в ManaJet): гибкие права до уровня фонда и статьи расхода персонально; приглашение по ссылке; журнал аудита всех финансовых операций (History с типами событий); заявка привязана к посту — права на подачу определяются назначением на пост.

Реализация: Supabase Auth + RLS-политики PostgreSQL (наработки миграций 001/002 актуализировать под модель v2.0).

## 4. Функциональные модули

### 4.1 Финансовое планирование (приоритет №1)

**4.1.1 Периоды.** Финансовая неделя чт–ср (настраиваемо), автосоздание следующего периода (autocreate как в FpPlan), статусы открыт/планирование/закрыт.

**4.1.2 Доходы.** Дерево видов дохода = текущее дерево прототипа (D-коды, папки-точки). Операция дохода: вид, сумма, валюта, дата, **счёт ДС, способ оплаты, контрагент (опционально, для банкетов — клиент), документ-основание, is_return**. Источники: ручной ввод, импорт, iiko (через outer_id), автосоздание из оплаты клиентского счёта. Автостатистика дохода.

**4.1.3 Схемы распределения.** Три этапа (выручка → маржинальный → скорректированный). Правило: вид дохода → фонд, этап, процент и/или фикс, приоритет. **Этап — свойство правила, не фонда** (ФД6 пополняется с двух этапов). Своя схема на каждый вид дохода. Редактируемые проценты в Директиве с сохранением скорректированной схемы периода (как в прототипе).

> ⚠️ ОТМЕНЕНО решением заказчика (см. docs/funds-spec.md §10–12): фонд одноэтапный, этап — свойство фонда (`funds.stage`), «домашний этап» в Директиве = дефолтное правило `distribution_rules`; смена этапа фонда — RPC `fp_set_fund_stage`. Правило «ФД6 пополняется с двух этапов» больше не действует.

**4.1.4 Фонды.** Текущая структура ФД1–ФД9/1 — стартовый справочник. Типы рабочий/накопительный; закрытые фонды (ФД5 Учредители, ФД6 Резервы, ФД7 Строительный — доступ по списку); папки; операции: пополнение, перемещение, заём с возвратом (реализовано в прототипе — перенести логику), обмен валют. Балансы — производная леджера (fp_register), запрет минуса на уровне БД.

**4.1.5 Заявки.** Формат ManaJet: пост заявителя, статья РД, фонд, способ оплаты, плановая сумма, валюта, период, **ЗРС: данные / ситуация / решение**, вложения (фото счёта). Статусы: подана → на планировании → одобрена/отклонена → оплачена. Группировка по отделениям оргсхемы в Директиве (как в прототипе). Оплата заявки (PurchaseOrderPayment): бухгалтер указывает счёт ДС, дату, сумму факта. Комментарии, уведомления.

**4.1.6 Счета поставщиков.** Контрагент, номер, дата, срок, статья, точка, ассортимент (StockCard-позиции), вложения. **Раздельные периоды одобрения и оплаты.** Панели: счета / одобрения / оплаты. Повторяющиеся счета (аренда, коммуналка). Возвраты поставщику. Приоритет одобрения счетов над заявками в Директиве. Печать/генерация из шаблона. **Этап 3: распознавание накладной из фото (аналог StockDocumentSmartSync).**

**4.1.7 Счета клиентам.** Банкетный счёт: клиент, мероприятие, зал/точка, дата, сумма, график оплат, частичные оплаты (InvoicePayment), статус долга (реализовано в прототипе). Планируемые счета = брони будущих дат, связь с CRM-воронкой. Генерация договора/счёта по шаблону (DocumentAttachmentGenerator) с реквизитами. Оплата клиентского счёта порождает операцию дохода.

**4.1.8 Контроль средств.** Счета ДС (FpAsset): кассы точек, банковские счета, карты, эквайринг; папки по точкам; валюта счёта. Расчётный остаток — из реестра (доходы на счёт − оплаты со счёта ± перемещения). Сверка: бухгалтер вводит факт, система показывает расхождение (снимок на момент сверки). Траты вне ФП — отдельный тип операции с подсветкой. Панель «Подробно» по счёту = выписка из реестра.

**4.1.9 Реестр операций.** Единая лента всех операций ФП с типом (доход, оплата заявки, оплата счёта, перемещение, заём, возврат, корректировка, вне ФП), фильтрами по периоду, контрагенту, счёту ДС, способу оплаты, точке. Источник истины для балансов фондов и счетов.

**4.1.10 Директива.** Экран прототипа как основа: выбор недели, доход периода, три уровня с расчётом и одобрением, корректировка процентов, панель заявок по отделениям, одобрение счетов (приоритетно), контроль остатков. Закрытие периода блокирует операции; протокол Директивы — автодокумент.

**4.1.11 Безокладная ЗП (этап 2).** Логика прототипа: баллы × коэффициент состояния (Власть 1.3, Изобилие 1.15, Норма 1.0, ЧП 0.85, Опасность 0.7, Несуществование 0.9), стоимость балла = ФОТ ÷ Σ баллов, ФОТ не превышается. Состояние подтягивается из статистики поста автоматически. Ведомости (SheetOfAccounts): аванс, остаток, удержания, экспорт.

**4.1.12 Отчёты.** ДДС по доходам/расходам и по фондам, P&L, сравнение точек (доход/расход/прибыль/ФОТ% — макет в прототипе), поставщики и клиенты (ДДС + отгрузки), фильтры по периодам и точкам, сравнение периодов, выгрузка в Excel.

### 4.2 Статистики (приоритет №1)
Перенос логики прототипа + модель ManaJet: типы (обычная, групповая, формульная, автоматическая из ФП/Потока, внешняя), периоды (неделя/месяц), **квоты** (план vs факт на графике), флаг invert, автоопределение состояния по тренду, ручное назначение состояния, привязка к посту/сотруднику/точке, ИЦО-доска компании и каждой точки, права доступа.

### 4.3 Оргсхема
7 отделений (нумерация прототипа: 7-1-2-3-4-5-6), посты с флагами is_main/is_holder/is_executive, несколько шляп на человека, ЦКП поста, статистики поста, подчинённость, вакансии, печать. Коммуникационный центр: **ЗРС-сообщения от поста к посту** (MsgCsw) — «кому писать по какому вопросу».

### 4.4 Шляпы (должностные папки)
Библиотека по отделениям/постам (HAT_LIB прототипа как стартовая структура), редактор, версии, отметка «изучил», привязка к постам.

### 4.5 Задачи
Статусная модель ManaJet: подготовлена → не уведомлён → выполняется → выполнена → **одобрена постановщиком** / невыполнима. Ответственный = пост или человек, сроки, чек-листы, вложения, комментарии, повторяющиеся задачи, подзадачи. Виды: список, календарь, Гант. Боевое планирование — план дня, привязанный к задачам и статистикам (макет в прототипе).

### 4.6 Управление проектами (этап 3)
Адмшкала: Цель → Замыслы → Оргполитика → Планы → Программы → Проекты → Идеальная картина → ЦКП → Статистики. Программа «100 точек к 2030», шаблон-чеклист открытия точки.

### 4.7 Поток (этап 3)
Модель ManaJet: доска → этапы (списки) → карточки → шаги; привязка доски к периоду ФП; доп. поля; авто-действия; статистики из карточек. Типовые доски: «Воронка банкетов» (этапы из CRM-прототипа: заявка → показ → предложение → договор → выигран + мероприятие → доплата), «Открытие точки», «Найм», «Стройка».

### 4.8 Маршрутные формы (этап 2)
Шаблоны цепочек согласования (RoutingFormTemplate): последовательность шагов по постам с доп. полями на шаг. Применение: согласование договоров на туй, приём сотрудника, ввод новой точки.

### 4.9 CRM (этап 2)
Единая сущность контрагента (клиент/поставщик), контакты, категории, адреса, вложения, дни рождения; история (счета, задачи в карточке); воронка; рассылки (Massmail) — SMS/Telegram: напоминания о доплате, поздравления.

### 4.10 Сотрудники и Личный кабинет
Сотрудники: карточка, посты, баллы ЗП, права, доп. поля, архив. ЛК: мои статистики, задачи, заявки со статусами, сообщения, боевое планирование, подача ЗРС.

### 4.11 Модуль «Ресторан» — решение по судьбе
В прототипе есть операционный модуль (заказы, столы, меню, склад, смены). Это территория iiko: дублировать POS-функциональность нецелесообразно. Решение для ТЗ: модуль в продукт **не переносится**; вместо него — интеграция iiko (выручка → операции дохода с привязкой к счёту ДС, номенклатура → StockCard, закрытие смены → автостатистика). Экраны прототипа сохранить как дизайн-референс на случай будущего собственного POS для малых точек. *(Если не согласен — обсуждаем, это моё предложение, в ТЗ помечено как решение заказчика.)*

## 5. Мультиточечность (ключевая адаптация — в ManaJet этого нет)
Сущность «Точка»: город, тип (туйхона/ресторан/кафе), статус (работает/стройка/ремонт), управляющий. Привязка к точке: виды дохода (уже в дереве прототипа), статьи расхода, фонды (опционально), счета ДС, заявки, счета, статистики, задачи. Переключатель «вся сеть / точка» во всех списках и отчётах. Сравнительный дашборд точек.

## 6. Интеграции
1. **iiko** (этап 3, приоритет): выручка смен → FpIncome-аналог через outer_id; номенклатура → StockCard.
2. **Telegram-бот** (этап 1): уведомления о заявках, ввод статистик, напоминания.
3. **Excel/Google Sheets**: импорт/экспорт ключевых разделов.
4. **Открытое REST API** (этап 3) по образцу ManaJet: API-ключи, outer_id во всех сущностях, Swagger-документация.
5. Банки Таджикистана/Alif — выписки для сверки (этап 3-4, по возможности).

## 7. Нефункциональные требования
- Тёмная/светлая тема прототипа (THEMES переносится), RU-интерфейс, i18n под TJ заложить (переключатель уже в каркасе).
- Mobile-first для ЛК, заявок, статистик, ввода доходов (управляющие работают с телефона).
- Списки до 100 000 записей, отклик < 300 мс; пагинация skip/take как в ManaJet.
- HTTPS, 2FA для финансовых ролей, ежедневный бэкап, аудит финансовых операций, транзакционность (балансы только через леджер, запрет минуса на уровне БД).
- Доступность 99,5%.

## 8. Технологический стек (уточнение v2.0)
- **Frontend:** React + TypeScript + Tailwind; перенос компонентов и стилей YakkasaroyFinance.jsx; Recharts.
- **Backend/БД:** **Supabase** — PostgreSQL + RLS (права), Auth (роли, инвайты), Storage (вложения), Edge Functions (Директива, расчёт ЗП, Telegram-бот, iiko-синк), Realtime (статусы заявок).
- Регион хостинга — Frankfurt (ближайший к Таджикистану по латентности).

## 9. Этапы
**Этап 1 — MVP финконтура (2–3 мес.):** роли/RLS, точки, периоды, доходы (со счетами ДС и способами оплаты), схемы распределения, фонды + реестр, заявки в ЗРС-формате от постов, Контроль средств со сверкой, Директива, автостатистики дохода, Telegram-уведомления. *Приёмка: неделя сети проведена в системе от ввода доходов до закрытия Директивы, остатки сходятся.*
**Этап 2 — Управление (2 мес.):** счета поставщиков (2 периода) и клиентам (частичные оплаты, договоры по шаблонам), статистики с квотами и ИЦО, задачи со статусом одобрения, оргсхема + ЗРС-коммуникации, шляпы, маршрутные формы, ЗП, отчёты.
**Этап 3 — Процессы и рост (2 мес.):** Поток, проекты/адмшкала, CRM + рассылки, iiko, распознавание накладных, открытое API.
**Этап 4 — Масштабирование:** TJ-локализация, банки, PWA, аналитика 20+ точек.

## 10. Что изменилось относительно ТЗ v1.0
1. Заявка: добавлен обязательный ЗРС-формат (данные/ситуация/решение) и подача от поста оргсхемы.
2. Операция дохода: добавлены счёт ДС, способ оплаты, контрагент, документ, is_return — Контроль средств теперь производная, а не параллельный учёт.
3. Счёт поставщика: два раздельных периода (одобрения и оплаты).
4. Этап распределения перенесён из фонда в правило распределения (кейс ФД6 из прототипа).
   > ⚠️ ОТМЕНЕНО решением заказчика (см. docs/funds-spec.md §10–12): фонд одноэтапный, этап — свойство фонда (`funds.stage`), смена этапа — RPC `fp_set_fund_stage`. Пункт сохранён как история изменений v1→v2.
5. Добавлены: единый Реестр операций, квоты статистик, статус «одобрена» у задач, маршрутные формы, ЗРС-сообщения/коммуникационный центр, единый контрагент CRM, outer_id и архивирование во всех сущностях, генератор документов, распознавание накладных из фото.
6. Зафиксированы реальные справочники из прототипа (ФД, РД, D-коды, отделения, состояния и коэффициенты ЗП) как стартовые данные системы.
7. Модуль «Ресторан» исключён из разработки в пользу интеграции iiko (решение на подтверждении заказчика).
8. Стек зафиксирован: Supabase вместо «NestJS/Django на выбор подрядчика».

## Приложение А. Глоссарий
ФП — финансовое планирование; ФРС — финансовое распределение средств; Фонд — запас средств на цель; Директива — еженедельное распределение и рассмотрение заявок; **ЗРС — законченная работа сотрудника (данные → ситуация → решение), формат заявки и сообщения руководителю**; Статистика — измеримый показатель поста; Квота — план по статистике на период; ЦКП — ценный конечный продукт; Шляпа — должностная папка; ИЦО — информационный центр организации; Реестр — единая лента операций ФП; Туйхона — банкетный зал; Точка — заведение сети.
