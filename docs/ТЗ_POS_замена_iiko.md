# ТЗ — Собственный POS «Яккасарой» (замена iiko)

**Статус:** черновик на обсуждение (v0.1)
**Контекст:** заказчик решил **полностью отказаться от iiko** и заменить его собственной
системой, чтобы не зависеть от внешнего вендора. Это разворот решения ТЗ v2 §4.11
(«модуль Ресторан не переносим, его закроет интеграция iiko»). Настоящий документ —
ТЗ на собственный операционный контур (POS + кухня + склад), который встаёт **поверх
уже существующего бэк-офиса** `YakkasaroyFinance` (Supabase) и QR-меню `yakkasaroy-menu`.

> **Зафиксированные вводные от заказчика (24.06.2026):**
> 1. Менять iiko **целиком**, не только печать.
> 2. **Фискальный чек юридически НЕ обязателен** → POS заметно проще (нет сертифицированного
>    фискального регистратора и онлайн-передачи в налоговую). Печатаем обычный сервисный/
>    гостевой чек на термопринтере. *(Если статус изменится — см. §11 «Фискализация».)*
> 3. Начинаем с ТЗ, код — последующими этапами.

---

## 1. Цель и принцип

Заменить iiko на собственную систему, закрыв все его рабочие роли в зале, на кухне и на
складе, при этом:

- **Переиспользовать существующий бэк-офис, а не дублировать.** Поставщики, счета,
  фонды, ЗП, права, отчётность уже есть в `YakkasaroyFinance`. POS пишет в **те же**
  таблицы Supabase. Это главное преимущество перед «голым» сторонним POS.
- **Деньги — только через Реестр `fp_register`** (инвариант проекта). Закрытие смены POS
  создаёт операцию дохода → запись в Реестре, как любой другой доход. POS **не** ведёт
  параллельный денежный учёт.
- **Каждая точка — отдельно** (Душанбе · Худжанд · Панджакент), владелец видит консолидацию.
- **Мобайл-фёрст:** официант работает с планшета/телефона, кухня — с экрана. Адаптив
  обязателен (правило проекта).

### Что заменяем (роли iiko) и где это уже есть

| Роль iiko | Где в нашей системе |
|---|---|
| Бэк-офис: поставщики, накладные, ЗП, отчёты, права | **Уже есть** — `YakkasaroyFinance` |
| Онлайн/QR-меню для гостя | **Уже есть** — `yakkasaroy-menu` (доработать сохранение заказа) |
| Front официанта, столы, счёт, оплата | **Строим** — модуль POS (§4) |
| KDS + печать на кухню/бар | **Строим** — §5 (часть уже спроектирована в «Проекте печати») |
| Номенклатура, модификаторы, техкарты | **Строим** — §6 |
| Склад, списание по техкартам, себестоимость, инвентаризация | **Строим** — §7 |
| Закрытие смены → выручка | **Строим** связку POS → `fp_register` — §8 |
| Фискализация | Вне scope (см. §11) |

## 2. Архитектура

- **Стек:** существующий — React 18 + Vite + Supabase (PostgreSQL + RLS + Auth + Realtime
  + Edge Functions). Никакой второй экосистемы (Odoo/Samba) — только как референс.
- **Realtime** — нерв системы: заказ из зала мгновенно появляется на кухне; статус
  «готово» — у официанта; стоп-лист — у всех.
- **Edge Functions** — закрытие смены (расчёт и проводка дохода), выдача задания печати
  (CloudPRNT), списание склада по техкартам.
- **Печать** — по отдельному «Проекту печати» (ESC/POS, станции, очередь `print_jobs`).
  Здесь не дублируется, считается приложением к этому ТЗ.

### Клиенты (рабочие места)

| Клиент | Кто | Устройство | Где в коде |
|---|---|---|---|
| **Меню-гость** | гость | телефон по QR (публично) | `yakkasaroy-menu` `…/?table=N` |
| **POS-официант** | официант/кассир | планшет/телефон (по ссылке, логин) | `yakkasaroy-menu` `…/pos` |
| **KDS-кухня** | повар/бариста | экран/планшет на кухне (по ссылке, логин) | `yakkasaroy-menu` `…/kds` |
| **Менеджер точки** | управляющий | телефон/десктоп | `YakkasaroyFinance` |

### Решение по размещению (отвечает на §14 вопрос 4)

**POS-фронт и KDS живут в репозитории `yakkasaroy-menu` и доступны по ссылке** — так же,
как уже сделано меню гостя. Причина: в `yakkasaroy-menu` уже есть отлаженный поток
«список → карточка блюда → корзина (`CartSheet`) → оформление (`Checkout`)», и POS-официант
— это та же корзина плюс выбор стола, отправка на кухню и оплата. Не строим с нуля —
надстраиваем над существующим.

- **Две «двери» в одном приложении, разделение по URL + логину:**
  - `…/` (или `…/?table=N`) — **меню гостя, публично** (как сейчас);
  - `…/pos` — **POS персонала**: логин + роли `waiter`/`cashier`/`cook`;
  - `…/kds` — **экран кухни**.
  Цены/оплату/отправку на кухню анонимному гостю не показываем — только за логином.
- **Бэк-офис** (фонды, Реестр, поставщики, ЗП, отчёты) остаётся в `YakkasaroyFinance`.
- **Общая база — ОДИН проект Supabase на оба фронта.** Это обязательное условие: только так
  POS пишет заказы и выручку в общий `fp_register`. В `yakkasaroy-menu` (сейчас чистый
  фронт без Supabase) заводится Supabase-клиент и нужная часть типов (допустимое небольшое
  дублирование; общий пакет — на будущее).
- Маршрутизация: сейчас в `yakkasaroy-menu` роутера нет — добавляется лёгкий роутинг или
  переключение вида по URL (`/pos`, `/kds`).

#### Точка входа из финансов: «Ресторан» = ссылка-портал (не встроенный модуль)

Модуль **«Ресторан»** сейчас скрыт из сайдбара `YakkasaroyFinance` (`src/data/navigation.js`,
закомментирован в `MODULES`). Решение:

- Пункт «Ресторан» **возвращается в сайдбар**, но при клике **НЕ рендерит ресторанные
  экраны внутри финансов** — он **открывает полный POS по ссылке в новой вкладке**
  (адрес деплоя `yakkasaroy-menu`, путь `…/pos`). Это тот же приём, что уже применён для
  меню гостя — «доступно по ссылке».
- **Без iframe.** Полная изоляция приложений: финансы только хранят URL POS и делают
  переход (`window.open`).
- **POS-код в `YakkasaroyFinance` не заезжает** — экраны `src/modules/restaurant/*`
  остаются дизайн-референсом, а боевой POS строится в `yakkasaroy-menu`. Так
  `YakkasaroyFinance` **не утяжеляется** (ни складом, ни KDS, ни кассой в его бандле).
- Связь между приложениями — **только на уровне данных** (общий Supabase: заказы/смены →
  `fp_register`), а не на уровне кода.
- Адрес POS хранить в переменной окружения (напр. `VITE_POS_URL`), чтобы не хардкодить
  домен деплоя.

## 3. Роли и доступ (RLS)

Поверх существующих ролей проекта (`owner`, `fin_director`, `ops_director`,
`location_manager`, `accountant`, `employee`) добавляются операционные:

- **waiter** (официант) — открывает/ведёт свои столы, отправляет на кухню, принимает оплату.
- **cashier** (кассир) — оплата, открытие/закрытие смены, Z-отчёт.
- **cook** (повар/бариста) — видит KDS своей станции, меняет статус блюда.

Доступ ограничивается точкой через существующую `has_location_access()`. Новые таблицы —
**только с RLS** на родных функциях прав (правило проекта). Официант видит только заказы
своей смены/точки; повар — только задания своей станции.

## 4. Модуль POS-официант (front зала)

Базируется на готовых дизайн-референсах `src/modules/restaurant/RestOrders.jsx`,
`RestTables.jsx`, `RestMenu.jsx` — их «оживляем» на Supabase.

**Функции:**
- **Залы и столы** — карта столов, статус (свободен/занят/счёт/резерв), привязка к точке.
- **Открытие заказа** на столе (или приём заказа из QR-меню гостя в общий список стола).
- **Добавление позиций** из меню: категории → блюдо → **модификаторы** (без сахара,
  степень прожарки, размер), количество, комментарий повару.
- **Курсы/подачи** (опц.) — отправлять блюда волнами.
- **Отправка на кухню** → создаёт `print_jobs` по станциям и толкает KDS (Realtime).
- **Счёт:** разделить/перенести/объединить столы, скидка (с правом), сервисный сбор (опц.).
- **Пречек** (предварительный счёт гостю) — печать без закрытия.
- **Оплата:** наличные / QR (Ягона) / карта; частичная оплата и разделение по гостям.
- **Стоп-лист:** если на складе кончился ингредиент — блюдо помечается «нет в наличии»
  у официанта и в QR-меню.
- **Смена официанта/кассира:** открыть/закрыть, итог по смене.

**Статусы заказа** (уже намечены в референсе): `new → cooking → ready → served → paid`.

**Мобильная адаптация:** список столов — карточки; добавление позиций — крупные тапы;
корзина-счёт — нижний лист (как в `yakkasaroy-menu`).

## 5. KDS + печать на кухню

См. отдельный **«Проект: печать заказов на кухонный принтер без iiko»**. Кратко:

- Маршрутизация по **станциям печати** (`print_stations`): кухня / бар / кофе — по
  `menu_categories.station_id`. Одно «Отправить на кухню» может дать несколько тикетов.
- **KDS-экран** — список активных блюд станции в реальном времени; повар отмечает
  «взял в работу» / «готово»; «готово» прилетает официанту.
- **Печать** — ESC/POS: вариант A (CloudPRNT-принтер, без ПК на кухне) или вариант B
  (дешёвый принтер + локальный агент). Очередь `print_jobs` + повторы при сбое.
- KDS — страховка на случай, если принтер «зажевал»: заказ не теряется.

## 6. Номенклатура, модификаторы, техкарты

- **Меню** — единый источник и для QR-гостя, и для POS, и для расчёта себестоимости.
  Категории, блюда, цена, фото, station_id, признак «весовое», стоп-лист.
- **Модификаторы** — группы (обязательные/необязательные, одиночный/множественный выбор),
  влияние на цену и на списание (доп. сыр = +порция сыра со склада).
- **Техкарты (рецептуры)** — блюдо = список ингредиентов с нормой расхода (брутто/нетто).
  Нужны для §7 (списание и себестоимость). Поддержать полуфабрикаты (техкарта внутри
  техкарты).
- **Соглашения схемы проекта:** `outer_id` (на будущее для импортов), `is_archived`
  вместо удаления, привязка к валюте `currency_id`.

## 7. Склад и себестоимость

Самая «бухгалтерски тяжёлая» часть — но стыкуется с уже имеющимися счетами поставщиков.

- **Складские карточки** (ингредиенты/товары), единицы измерения, остатки по точкам
  (референс — `RestStock.jsx`).
- **Приход** — из накладной поставщика. **Связка:** накладная прихода ↔ счёт поставщика
  в `YakkasaroyFinance` (не дублировать ввод). Распознавание накладных — на будущее.
- **Списание при продаже** — Edge Function по факту продажи блюда списывает ингредиенты
  по техкарте (с учётом модификаторов).
- **Себестоимость (COGS)** — средневзвешенная по приходам; даёт маржу по блюду и по точке.
- **Инвентаризация** — сверка факт/учёт, акт расхождений.
- **Стоп-лист** — автоматически из нулевого остатка ключевого ингредиента.

> Это можно вынести в отдельный поздний этап: на старте POS может работать **без
> авто-списания** (только продажи и выручка), а склад/себестоимость подключить вторым
> заходом. Помечено в §10.

---

## 8. Связка с финансами (главное ради чего всё)

- **Закрытие смены** (Z-отчёт) формирует итог выручки по точке/способам оплаты и через
  Edge Function создаёт **операцию дохода** в `incomes` → запись в **`fp_register`**
  (с привязкой к счёту ДС и способу оплаты — инвариант №1 предметной области).
- Это ровно то, что ТЗ v2 планировал тянуть из iiko через `outer_id` — теперь **источник
  свой**. Поле `outer_id` сохраняем (id смены POS) для прослеживаемости.
- **Балансы фондов/счетов руками не трогаем** — их ведут триггеры Реестра. POS только
  поставляет факт дохода.
- Приход склада ↔ счета поставщиков; ФОТ официантов/поваров ↔ существующий модуль ЗП
  (в т.ч. ЗП по результату/баллам, если потребуется привязать к выручке смены).

## 10. Дорожная карта (этапы)

> Принцип внедрения: **не «большой взрыв»**. POS запускается на **одной пилотной точке
> параллельно с iiko**, месяц сверки, затем отключение iiko и тираж.

**Этап 1 — Заказы + кухня (MVP операционки).**
`orders/order_items/print_stations`, сохранение заказа из QR-меню, рабочее место официанта
(оживить `RestOrders/RestTables`), KDS + печать. → Можно работать в зале.

**Этап 2 — Меню/модификаторы + оплата + смена.**
Полноценное меню с модификаторами и стоп-листом, оплата (нал/QR/карта), смена кассира,
Z-отчёт. **Закрытие смены → доход в `fp_register`** (ключевая связка §8).

**Этап 3 — Склад + техкарты + себестоимость.**
Складские карточки, приход из накладных (связка со счетами поставщиков), списание по
техкартам, инвентаризация, маржа/себестоимость.

**Этап 4 — Паритет с iiko.**
Отчёты POS (выручка, средний чек, ABC по блюдам, списания), офлайн-режим кассы (см. §12),
лояльность/скидки, доставка/агрегаторы (по необходимости).

**Этап 5 — Параллельная работа и отключение iiko** на пилоте, затем тираж на 3 точки.

## 11. Фискализация (вне scope, но фиксируем)

По решению заказчика фискальный чек **не обязателен** → печатаем обычный гостевой/
сервисный чек. Если требование появится (изменение законодательства/проверка):
- интеграция с сертифицированным фискальным регистратором или допустимым фискальным ПО;
- это **отдельный модуль**, не ломающий модель данных (оплаты и смены уже есть).
Риск помечен, чтобы решение было осознанным.

## 12. Риски и нефункциональные требования

- **🟠 Офлайн-надёжность.** iiko работает при упавшем интернете; наша касса — облачная.
  Касса в зале **не должна вставать** при обрыве сети → локальная очередь операций и
  досинхронизация при восстановлении. Серьёзная инженерная задача (Этап 4 минимум, либо
  раньше для пилота).
- **🟠 Себестоимость/техкарты** — самая трудоёмкая бухгалтерская часть (Этап 3).
- **🟡 Оборудование** — весы для весовых блюд, денежный ящик, второй экран; по мере нужды.
- **Надёжность печати** — очередь `print_jobs` + KDS как страховка (§5).
- **Производительность** — Realtime на час пик (десятки столов × позиции); проектировать
  подписки по точке/станции, не «на всё сразу».
- **Обязательная мобильная адаптация и тёмная тема** — правило проекта, для всех экранов.

## 13. Definition of Done (по правилам проекта)

- Новый код — TypeScript; обращения к БД — через типизированный клиент.
- Изменения схемы — новым файлом миграции в `supabase/migrations/`, RLS на родных
  функциях прав, pgTAP-тест инварианта, перегенерация `database.types.ts`.
- Деньги — только через `fp_register` (инварианты держит БД).
- Перед PR — зелёные `typecheck`, `test`, `lint`, `build`; каждое изменение — отдельным PR.
- Мобильная адаптация — часть каждого изменения, не отдельная задача.
