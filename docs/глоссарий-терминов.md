## 1. Git и совместная работа над кодом

### Репозиторий (repository, «репа»)
**Что это.** Папка с твоим проектом, за которой ведётся история всех изменений.
Представь шкаф с документами, где каждый раз, когда ты что-то поправил, делается
«фотография» всего шкафа — и любую старую версию можно вернуть. Репозиторий хранит
и сам код, и всю эту историю «фотографий».

**3 примера на практике:**
- Весь проект «Yakkasaroy Finance» лежит в репозитории на GitHub — оттуда я его «клонирую» себе, чтобы работать.
- Когда нужно вернуться к версии «как было неделю назад», смотрим историю репозитория.
- Новый разработчик, чтобы начать работу, первым делом копирует (клонирует) репозиторий.

**3 примера в предложении:**
- «Я запушил изменения в репозиторий, проверь последнюю версию.»
- «В этом репозитории два модуля: финансы и меню.»
- «Создай отдельный репозиторий под мобильное приложение.»

---

### Git
**Что это.** Программа, которая и ведёт ту самую историю изменений в репозитории.
Git — это «машина времени» для кода: запоминает, кто, что и когда поменял, и
позволяет нескольким людям править один проект, не затирая работу друг друга.

**3 примера на практике:**
- Git хранит, что именно изменилось в каждом файле между версиями.
- Через Git два разработчика правят разные файлы одновременно и потом объединяют работу.
- Git позволяет «откатить» неудачное изменение к рабочему состоянию.

**3 примера в предложении:**
- «Зафиксируй это в Git, чтобы не потерять.»
- «Git показывает, что ты изменил три строки в этом файле.»
- «Это стандартный Git-процесс, ничего особенного.»

---

### Коммит (commit)
**Что это.** Одна сохранённая «фотография» изменений с подписью-комментарием.
Это как поставить точку: «вот это законченный кусочек работы, запоминаем его под
названием». У каждого коммита есть автор, дата и описание, что было сделано.

**3 примера на практике:**
- После того как починил ошибку в расчёте фонда — делаешь коммит с описанием «исправлен расчёт ФД4».
- История коммитов показывает, как проект развивался день за днём.
- Если что-то сломалось, находим коммит, после которого появилась проблема, и откатываемся.

**3 примера в предложении:**
- «Я сделал коммит с понятным сообщением.»
- «В этом коммите слишком много изменений сразу, лучше разбить.»
- «Последний коммит сломал сборку — откатываем.»

---

### Ветка (branch)
**Что это.** Отдельная «параллельная вселенная» проекта, где можно спокойно
экспериментировать, не трогая основную версию. Как черновик рядом с чистовиком:
поработал в черновике, всё проверил — и только потом перенёс в чистовик.
Основная ветка обычно называется `main`.

**3 примера на практике:**
- Для нового глоссария я создал ветку `claude/technical-terms-glossary-...`, чтобы не задеть рабочий `main`.
- Каждую новую функцию делают в своей ветке, а потом сливают в основную.
- Если в ветке всё пошло не так, её можно просто удалить — `main` не пострадает.

**3 примера в предложении:**
- «Я работаю в отдельной ветке, основной код не трогаю.»
- «Переключись на ветку `main`, чтобы увидеть актуальную версию.»
- «Создай ветку под эту задачу.»

---

### Пуш (push)
**Что это.** Отправить свои коммиты с компьютера на сервер (GitHub), чтобы они
стали видны другим. Локально ты насохранял «фотографий» — пуш выкладывает их в
общее облако.

**3 примера на практике:**
- Закончил правки → пушишь ветку на GitHub, чтобы заказчик увидел изменения.
- Без пуша твоя работа остаётся только на твоём компьютере.
- После пуша автоматически запускается проверка и сборка превью.

**3 примера в предложении:**
- «Я запушил изменения, посмотри на GitHub.»
- «Не забудь запушить, иначе я не увижу твою работу.»
- «Пуш не прошёл из-за сети, повторю.»

---

### Pull / Fetch (подтянуть изменения)
**Что это.** Обратная операция к пушу: забрать с сервера то, что туда положили
другие, и обновить свою копию. `Fetch` — просто посмотреть, что нового;
`Pull` — посмотреть и сразу влить к себе.

**3 примера на практике:**
- Перед началом работы делаешь pull, чтобы работать на самой свежей версии.
- Fetch показывает, что в `main` появились новые коммиты от коллеги.
- После pull твоя локальная копия совпадает с серверной.

**3 примера в предложении:**
- «Сделай pull перед тем как начинать, там есть новые правки.»
- «Я зафетчил ветку, но ещё не сливал.»
- «Подтяни изменения, иначе будет конфликт.»

---

### Pull Request (PR, «пул-реквест»)
**Что это.** Официальная заявка: «я сделал изменения в своей ветке, прошу влить
их в основную». PR — это место, где изменения обсуждают, проверяют и одобряют
перед тем как они станут частью продукта. В нашем проекте **каждое изменение
оформляется отдельным PR в `main`**.

**3 примера на практике:**
- Сделал доработку → открываешь PR → заказчик смотрит превью → одобряет → изменение уходит в продакшен.
- В PR видно построчно, что именно изменилось.
- Бот Vercel под PR присылает ссылку на превью-версию приложения.

**3 примера в предложении:**
- «Я открыл PR, проверь превью-ссылку.»
- «Не делаем несколько PR сразу — ждём, пока смержат текущий.»
- «В этом PR три файла изменены.»

---

### Merge (слияние, «смержить»)
**Что это.** Объединить изменения из одной ветки в другую. Когда PR одобрен,
его «мержат» в `main` — и черновик становится частью чистовика.

**3 примера на практике:**
- Заказчик одобрил PR → нажимает «Merge» → продакшен на Vercel обновляется.
- Слияние двух веток объединяет работу двух разработчиков.
- Если две ветки правили одну строку — при мерже возникает «конфликт», его нужно разрешить вручную.

**3 примера в предложении:**
- «Как смержишь PR — я возьмусь за следующую задачу.»
- «При мерже возник конфликт в одном файле.»
- «Эту ветку уже смержили в `main`.»

---

### Clone (клонировать)
**Что это.** Скачать полную копию репозитория (со всей историей) себе, чтобы с
ним работать. Происходит один раз — в начале.

**3 примера на практике:**
- В облачной сессии репозиторий клонируется заново при старте контейнера.
- Новый разработчик клонирует проект и сразу видит весь код и историю.
- Клонируешь репозиторий → ставишь зависимости → запускаешь приложение.

**3 примера в предложении:**
- «Склонируй репозиторий и запусти `npm install`.»
- «Я работаю на свежем клоне проекта.»
- «Клонирование занимает пару секунд.»

## 2. Хостинг, деплой и окружения

### Деплой (deploy, «выкатить»)
**Что это.** Опубликовать приложение так, чтобы оно стало доступно в интернете
по адресу. Написать код мало — его нужно «развернуть» на сервере, чтобы им можно
было пользоваться через браузер.

**3 примера на практике:**
- После мержа PR в `main` Vercel автоматически делает деплой — продакшен обновляется.
- Каждый PR получает свой превью-деплой для проверки.
- Если деплой упал — приложение не обновилось, смотрим логи сборки.

**3 примера в предложении:**
- «Деплой прошёл успешно, можно открывать приложение.»
- «Это превью-деплой, не продакшен.»
- «После мержа автоматически выкатится новая версия.»

---

### Vercel
**Что это.** Сервис (хостинг), на котором живёт наше приложение. Он сам собирает
код, публикует его в интернете и для каждого PR делает отдельную превью-версию.
Можно думать о нём как о «типографии», которая печатает рабочую версию сайта из
кода.

**3 примера на практике:**
- Продакшен «Yakkasaroy Finance» работает на Vercel и привязан к ветке `main`.
- Vercel-бот присылает короткую превью-ссылку под каждым PR.
- Переменные окружения (ключи к базе) задаются в панели Vercel, а не в коде.

**3 примера в предложении:**
- «Открой превью-ссылку от Vercel.»
- «Vercel сам пересоберёт проект после мержа.»
- «Ключи лежат в настройках Vercel.»

---

### Превью (preview)
**Что это.** Временная рабочая версия приложения для конкретного изменения, чтобы
посмотреть результат **до** публикации в продакшен. Как тестовый показ перед
премьерой.

**3 примера на практике:**
- Заказчик открывает превью-ссылку и проверяет новую функцию, не рискуя «боевой» версией.
- На превью можно нажать кнопки, заполнить формы — всё работает по-настоящему.
- Если на превью что-то не так — правим, не трогая продакшен.

**3 примера в предложении:**
- «Дай короткую превью-ссылку, я проверю.»
- «На превью кнопка работает корректно.»
- «Это превью под конкретный PR.»

---

### Продакшен (production, «прод», «боевая версия»)
**Что это.** Настоящая, живая версия приложения, которой пользуются реальные люди.
Противоположность черновику/превью. Ошибки на проде видят все пользователи,
поэтому туда выкатывают только проверенное.

**3 примера на практике:**
- Продакшен обновляется только после мержа PR в `main`.
- На проде работают настоящие данные ресторанов, а не тестовые.
- Перед выкаткой на прод всё проверяют на превью.

**3 примера в предложении:**
- «На проде эта ошибка видна всем — срочно правим.»
- «Не экспериментируй на продакшене.»
- «Изменение ушло в прод после мержа.»

---

### Окружение / переменные окружения (environment, env)
**Что это.** «Настройки снаружи кода» — секретные ключи и параметры, которые не
вписывают в код, а подкладывают отдельно. Например, ключ доступа к базе данных.
Это как не писать пароль от сейфа прямо на двери, а хранить его отдельно.

**3 примера на практике:**
- `VITE_SUPABASE_URL` и `VITE_SUPABASE_KEY` — переменные окружения для подключения к базе.
- На Vercel ключи задаются в панели, локально — в файле `.env` (он не попадает в Git).
- Разные окружения (тест и прод) используют разные ключи к разным базам.

**3 примера в предложении:**
- «Ключи лежат в переменных окружения, не в коде.»
- «Добавь эту переменную в `.env`.»
- «На продакшен-окружении другой ключ к базе.»

## 3. База данных и Supabase

### База данных (БД, database)
**Что это.** Организованное хранилище всех данных приложения: пользователи,
фонды, доходы, расходы, заказы. В отличие от обычного файла, база умеет быстро
искать, фильтровать, связывать данные и не терять их. Представь огромную умную
картотеку.

**3 примера на практике:**
- Все доходы и расходы ресторанов хранятся в базе данных, а не в коде.
- Когда ты открываешь экран «Фонды», приложение запрашивает суммы из базы.
- База следит, чтобы баланс фонда не ушёл в минус.

**3 примера в предложении:**
- «Эти данные приходят из базы.»
- «Сохрани доход в базу данных.»
- «В базе уже 33 таблицы.»

---

### Supabase
**Что это.** Готовый «движок» базы данных и бэкенда «под ключ». Даёт сразу
несколько вещей: саму базу (PostgreSQL), вход по логину/паролю (Auth), хранилище
файлов (Storage), мини-программы на сервере (Edge Functions) и обновления в
реальном времени (Realtime). Вместо того чтобы писать всё это с нуля, мы берём
Supabase.

**3 примера на практике:**
- Финансовый модуль уже ходит за данными в Supabase, а не в моки.
- Вход в приложение (кто ты, какая у тебя роль) проверяется через Supabase Auth.
- Изменения схемы базы применяются через Supabase миграциями.

**3 примера в предложении:**
- «Этот модуль уже переведён на Supabase.»
- «Supabase сам проверяет права доступа через RLS.»
- «Подключи коннектор Supabase, чтобы применить миграцию.»

---

### PostgreSQL (Postgres)
**Что это.** Конкретная, очень надёжная и популярная программа-база данных,
которая работает «внутри» Supabase. Когда говорят «наша база» — технически это
PostgreSQL.

**3 примера на практике:**
- Таблицы фондов и доходов хранятся в PostgreSQL.
- PostgreSQL умеет считать суммы и проверять правила прямо в базе (триггерами).
- Запросы к данным пишутся на языке SQL, который понимает PostgreSQL.

**3 примера в предложении:**
- «Это стандартная возможность PostgreSQL.»
- «Запрос к Postgres вернул 12 строк.»
- «PostgreSQL держит инварианты на уровне базы.»

---

### Таблица (table)
**Что это.** Данные в базе хранятся в таблицах — как в Excel: строки и столбцы.
Одна таблица = один тип сущности. Например, таблица «фонды»: каждая строка —
один фонд, столбцы — название, баланс, этап.

**3 примера на практике:**
- Таблица `funds` хранит все фонды (ФД1–ФД9).
- Таблица `incomes` — все операции дохода.
- Таблица `profiles` — пользователи и их роли.

**3 примера в предложении:**
- «Добавь новую колонку в таблицу доходов.»
- «В этой таблице каждая строка — один счёт.»
- «Свяжи таблицу заявок с таблицей фондов.»

---

### Схема (schema)
**Что это.** Общая структура базы: какие есть таблицы, какие у них столбцы, как
они связаны и какие правила действуют. Схема — это «чертёж» базы, а данные —
то, что в неё налито.

**3 примера на практике:**
- Финансовый контур — это 33 таблицы плюс правила: всё это «схема».
- Изменение схемы (новая таблица/колонка) делается отдельной миграцией.
- Перед работой со схемой сверяемся со свежей миграцией.

**3 примера в предложении:**
- «Это изменение схемы — нужна миграция.»
- «Схема базы описана в `supabase/README.md`.»
- «Не меняй схему напрямую, только через миграцию.»

---

### Миграция (migration)
**Что это.** Пошаговое, записанное изменение структуры базы. Вместо того чтобы
«молча» поправить базу руками, мы пишем файл-инструкцию: «добавить такую-то
таблицу/колонку». Это даёт историю изменений базы — как коммиты, но для схемы.

**3 примера на практике:**
- Нужна новая колонка → создаём новый файл миграции в `supabase/migrations/`.
- Уже применённые миграции не редактируют — только добавляют новые поверх.
- Миграции применяю я через Supabase MCP, заказчик SQL руками не запускает.

**3 примера в предложении:**
- «Это требует отдельной миграции.»
- «Применил миграцию, схема обновилась.»
- «Сверься с последней миграцией перед изменениями.»

---

### RLS (Row Level Security, «защита на уровне строк»)
**Что это.** Правила, которые решают, **кто какие строки** в таблице может видеть
и менять. Не «есть доступ к таблице или нет», а тоньше: бухгалтер видит свою
точку, владелец — все. Эти правила живут в самой базе, поэтому их нельзя обойти
через приложение.

**3 примера на практике:**
- Менеджер точки в Душанбе видит доходы только своей точки — это делает RLS.
- Владелец видит консолидированную картину по всем точкам — тоже через RLS.
- Новая таблица заводится сразу с RLS-политиками на функциях прав.

**3 примера в предложении:**
- «Доступ к этим строкам ограничен через RLS.»
- «Добавь RLS-политику для новой таблицы.»
- «RLS не пустит чужого пользователя к данным другой точки.»

---

### Триггер (trigger)
**Что это.** Действие, которое база делает **автоматически** при каком-то
событии. Как датчик движения: записали доход → база сама создала запись в Реестре
и пересчитала баланс фонда. Нам не нужно это делать вручную.

**3 примера на практике:**
- Вставка дохода в `incomes` триггером сама создаёт запись в Реестре `fp_register`.
- Триггер не даёт балансу фонда уйти в минус.
- Триггер блокирует операции в уже закрытом периоде.

**3 примера в предложении:**
- «Баланс пересчитывается триггером, руками не пишем.»
- «Этот триггер запрещает минус на счёте.»
- «Добавь триггер, который ведёт историю изменений.»

---

### Функция (в базе) / RPC
**Что это.** Кусочек логики, который хранится и выполняется прямо в базе, и его
можно вызвать по имени из приложения. RPC — это «вызвать такую функцию базы из
кода». Например, «смени этап фонда» — одна функция, которая делает всё правильно
и согласованно.

**3 примера на практике:**
- `fp_set_fund_stage` — функция, которая корректно меняет этап фонда.
- Функции прав (`my_role`, `is_fin_admin`) определяют роль и доступ пользователя.
- Вместо сложной логики на клиенте вызываем RPC — база делает всё атомарно.

**3 примера в предложении:**
- «Смену этапа делает RPC `fp_set_fund_stage`.»
- «Эта функция возвращает роль текущего пользователя.»
- «Вызови RPC вместо ручных правок балансов.»

---

### Реестр / леджер (ledger, `fp_register`)
**Что это.** Единая лента всех денежных операций — «источник истины» про деньги.
Балансы фондов и счетов не хранятся отдельно «как есть», а **высчитываются** из
этой ленты. Как банковская выписка: сальдо — это итог всех движений, а не
отдельно вписанное число.

**3 примера на практике:**
- Любое движение денег (доход, расход, оплата) попадает в `fp_register`.
- Баланс фонда — это производная от записей Реестра, а не отдельная цифра.
- Запрет ухода в минус и блокировка закрытого периода держатся на уровне Реестра.

**3 примера в предложении:**
- «Деньги проводим только через Реестр.»
- «Баланс — производная леджера, руками не правим.»
- «Эта операция уже есть в Реестре.»

---

### Баланс (balance)
**Что это.** Текущая сумма «остатка» — на фонде, счёте, в кассе. В нашем проекте
баланс не вписывают вручную: он вычисляется из ленты операций (Реестра)
автоматически.

**3 примера на практике:**
- Баланс фонда ФД4 = сумма всех его поступлений минус расходы по Реестру.
- Триггер не даёт списать больше, чем есть на балансе.
- На дашборде показываем балансы всех фондов одним списком.

**3 примера в предложении:**
- «Проверь баланс фонда перед списанием.»
- «Баланс пересчитался автоматически после дохода.»
- «Нельзя уйти в отрицательный баланс.»

---

### Индекс (index)
**Что это.** Способ ускорить поиск в таблице — как алфавитный указатель в конце
книги. Без индекса база читает всю таблицу подряд; с индексом — сразу находит
нужное.

**3 примера на практике:**
- Индекс по `location_id` ускоряет выборку операций конкретной точки.
- При медленных запросах добавляют индекс на часто фильтруемую колонку.
- Индексы важны, когда в таблице много строк.

**3 примера в предложении:**
- «Запрос тормозит — добавим индекс.»
- «По этой колонке уже есть индекс.»
- «Индекс ускоряет поиск, но чуть замедляет вставку.»

---

### Enum (перечисление)
**Что это.** Тип данных с фиксированным списком допустимых значений. Например,
роль пользователя может быть только из набора: `owner`, `accountant`, `employee`
и т.д. База не даст вписать что-то вне списка — это защита от опечаток.

**3 примера на практике:**
- Роли пользователей заданы как enum — нельзя придумать роль «директор2».
- Состояния статистик ХМС (Норма, Опасность, …) удобно хранить как enum.
- Способ оплаты (наличные/карта/перевод) — тоже enum.

**3 примера в предложении:**
- «Это значение из enum, других быть не может.»
- «Добавь новое значение в enum ролей.»
- «Enum не пропустит недопустимый статус.»

---

### Constraint (ограничение)
**Что это.** Правило-«нельзя» на уровне базы, которое защищает данные от
бессмыслицы. Например: «баланс не может быть отрицательным» или «у дохода
обязательно должна быть точка». База просто откажется сохранять то, что нарушает
ограничение.

**3 примера на практике:**
- Ограничение «не минус» защищает балансы фондов.
- Ограничение «обязательное поле» не даёт сохранить доход без указания точки.
- Уникальное ограничение не даёт завести два одинаковых кода фонда.

**3 примера в предложении:**
- «Это нарушает ограничение базы.»
- «Поставь constraint на уникальность кода.»
- «Ограничение не пустило сохранение.»

---

### Auth (аутентификация, вход)
**Что это.** Система «кто ты такой»: вход по логину и паролю, проверка личности.
Supabase Auth выдаёт пользователю «пропуск» (сессию), по которому приложение
понимает, кто вошёл и что ему можно.

**3 примера на практике:**
- При входе в приложение Supabase Auth проверяет логин/пароль и заводит сессию.
- По данным Auth подтягивается профиль пользователя и его роль.
- Если сессия истекла — пользователя просят войти заново.

**3 примера в предложении:**
- «Вход обрабатывает Supabase Auth.»
- «После аутентификации тянем профиль и роль.»
- «Без авторизации экран недоступен.»

---

## 4. Фронтенд: из чего собран интерфейс

### Фронтенд / бэкенд (frontend / backend)
**Что это.** Фронтенд — это то, что пользователь **видит и трогает**: кнопки,
формы, таблицы в браузере. Бэкенд — то, что **за кулисами**: база, правила,
сервер. Официант (фронтенд) принимает заказ, кухня (бэкенд) готовит.

**3 примера на практике:**
- Экраны фондов и доходов — это фронтенд (React).
- Хранение и правила движения денег — бэкенд (Supabase/PostgreSQL).
- Фронтенд запрашивает данные у бэкенда и показывает их пользователю.

**3 примера в предложении:**
- «Это чисто фронтенд-задача — поправить вид таблицы.»
- «Логику считаем на бэкенде, не на клиенте.»
- «Фронтенд просто отображает то, что вернул бэкенд.»

---

### React
**Что это.** Библиотека, на которой построен наш интерфейс. Главная идея —
интерфейс собирается из переиспользуемых «кирпичиков» (компонентов), и когда
данные меняются, React сам перерисовывает нужные части экрана.

**3 примера на практике:**
- Каждый экран модуля — это React-компонент.
- Когда меняется баланс фонда, React сам обновляет карточку, не перезагружая страницу.
- Новые экраны мы пишем как React-компоненты в `src/modules/`.

**3 примера в предложении:**
- «Это React-компонент, лежит в модуле финансов.»
- «React сам перерисует список после изменения.»
- «Перепиши это на отдельный React-компонент.»

---

### Компонент (component)
**Что это.** Самостоятельный «кирпичик» интерфейса со своим видом и поведением,
который можно переиспользовать. Например, карточка фонда — компонент: один раз
описали, используем для всех фондов.

**3 примера на практике:**
- Карточка фонда — компонент, отображается списком для каждого фонда.
- Модальное окно (всплывающая форма) — отдельный компонент.
- Кнопка, таблица, шапка — всё это компоненты.

**3 примера в предложении:**
- «Вынеси это в отдельный компонент.»
- «Этот компонент переиспользуется на трёх экранах.»
- «Поправь компонент карточки, а не каждый экран.»

---

### State и props (состояние и свойства)
**Что это.** **State** — это «память» компонента, его текущее состояние (например,
открыта форма или нет, что введено в поле). **Props** — это данные, которые
компонент получает «снаружи», от родителя (например, какой именно фонд показать).
State компонент меняет сам; props ему передают.

**3 примера на практике:**
- Открыт ли модал — хранится в state.
- Какой фонд показать в карточке — приходит через props.
- При вводе суммы в поле меняется state, и экран обновляется.

**3 примера в предложении:**
- «Храни это в state компонента.»
- «Передай сумму через props.»
- «State сбрасывается при закрытии формы.»

---

### Hook (хук)
**Что это.** Специальная функция React, дающая компоненту «способности»: помнить
state, реагировать на события, обращаться к данным. Названия начинаются с `use`:
`useState`, `useTheme`, `useIsMobile`. Думай о хуке как о «розетке», через которую
компонент подключается к возможностям React.

**3 примера на практике:**
- `useState` даёт компоненту память (например, открыт ли модал).
- `useTheme` даёт доступ к цветам темы и текущему языку.
- `useIsMobile` подсказывает, рисовать ли мобильный вид.

**3 примера в предложении:**
- «Используй хук `useIsMobile` для мобильной вёрстки.»
- «Этот хук возвращает тему и стили.»
- «Хуки нельзя вызывать внутри условий.»

---

### Vite
**Что это.** Инструмент, который собирает наш проект и запускает локальный сервер
для разработки (с мгновенным обновлением при правках). «Сборщик и движок
разработки» — благодаря ему `npm run dev` поднимает приложение за секунду.

**3 примера на практике:**
- `npm run dev` запускает Vite на `http://localhost:5173`.
- `npm run build` — Vite собирает финальную версию в папку `dist/`.
- При сохранении файла Vite сам обновляет страницу в браузере.

**3 примера в предложении:**
- «Запусти Vite через `npm run dev`.»
- «Сборку делает Vite.»
- «Vite подхватил изменения мгновенно.»

---

### Tailwind / shadcn
**Что это.** **Tailwind** — способ оформлять элементы готовыми классами-«утилитами»
(`bg-panel`, `text-money`) вместо написания CSS с нуля. **shadcn** — набор готовых
красивых компонентов (кнопки, поля), которые можно взять и адаптировать под нашу
тему. Вместе они ускоряют создание аккуратного интерфейса.

**3 примера на практике:**
- Новые компоненты можно верстать классами Tailwind.
- Цвета в Tailwind-классах берутся из единой палитры темы.
- Готовый компонент из shadcn добавляют командой и подгоняют под тёмную тему.

**3 примера в предложении:**
- «Свёрстай новый экран на Tailwind.»
- «Возьми кнопку из shadcn и адаптируй под тему.»
- «Эти классы Tailwind ссылаются на палитру `C`.»

---

### Инлайн-стили / тема (inline styles, theme)
**Что это.** **Инлайн-стили** — оформление, заданное прямо у элемента. В нашем
проекте все цвета и стили собраны в одном месте — в **теме** (палитра `C`), чтобы
обе версии (тёмная/светлая) работали и не было «захардкоженных» цветов. Это как
единая фирменная палитра вместо случайных цветов на каждом экране.

**3 примера на практике:**
- Все цвета берутся из палитры `C`, а не вписываются числом в компонент.
- Общие стили лежат в `makeStyles` и используются как `st.имя`.
- При смене темы (dark/light) весь интерфейс перекрашивается из одного источника.

**3 примера в предложении:**
- «Не хардкодь цвет — возьми из палитры `C`.»
- «Добавь стиль в `makeStyles` и используй `st.card`.»
- «Проверь, что экран работает в обеих темах.»

---

### Адаптивность / `isMobile` (responsive)
**Что это.** Способность интерфейса нормально выглядеть и на компьютере, и на
телефоне. `isMobile` — подсказка «сейчас телефон», по которой мы перестраиваем
вёрстку (карточки вместо широких таблиц, формы в одну колонку). В нашем проекте
мобильная адаптация — **обязательная часть каждого изменения**.

**3 примера на практике:**
- На телефоне список фондов показываем карточками, а не таблицей со скроллом.
- Кнопки в шапке на узком экране переносятся на новую строку.
- Модальное окно на телефоне занимает всю ширину.

**3 примера в предложении:**
- «Проверь мобильный вид через `isMobile`.»
- «На телефоне таблица вылезает за край — поправь.»
- «Адаптив — обязательная часть задачи, не отдельная.»

## 5. Качество кода: типы, тесты, линтеры

### JavaScript / TypeScript (JS / TS)
**Что это.** **JavaScript** — язык, на котором работает наше приложение в браузере.
**TypeScript** — это JavaScript плюс «подписи к данным» (типы): он заранее
проверяет, что ты не передал текст туда, где ждали число. TS ловит ошибки **до**
запуска. В проекте новый чистый код пишем на TypeScript.

**3 примера на практике:**
- Чистые утилиты (`format.ts`, `funds.ts`) переведены на TypeScript.
- Тип `Database` описывает структуру базы, чтобы запросы были безопасны.
- Если перепутать число и строку, TypeScript подсветит ошибку сразу в редакторе.

**3 примера в предложении:**
- «Новый модуль пишем сразу на TypeScript.»
- «TypeScript ругается: здесь ждали число.»
- «Это обычный JavaScript-файл, типов пока нет.»

---

### Тип (type)
**Что это.** «Подпись», какого вида данные: число, строка, дата, объект с такими-то
полями. Типы — это как форма для документа: заранее ясно, что куда вписывать, и
нельзя положить не то.

**3 примера на практике:**
- Тип «Фонд» описывает, что у фонда есть код, название, баланс, этап.
- Функция форматирования денег принимает число и возвращает строку — это её тип.
- Типы базы (`database.types.ts`) генерируются из схемы автоматически.

**3 примера в предложении:**
- «Опиши тип для этой сущности.»
- «Тип не совпадает: ждали дату, передали строку.»
- «Перегенерируй типы после изменения схемы.»

---

### Unit-тест / Vitest (модульный тест)
**Что это.** Маленькая автоматическая проверка одного кусочка логики: «если на
вход дать это — должно получиться то». Тесты ловят поломки, когда ты что-то менял
и случайно сломал старое. Vitest — инструмент, который их запускает.

**3 примера на практике:**
- Тест проверяет, что суммы форматируются правильно (1234567 → «1 234 567»).
- Тест проверяет расчёт состояния статистики по правилам ХМС.
- Перед PR прогоняем все тесты — если красные, не выкатываем.

**3 примера в предложении:**
- «Напиши unit-тест на эту функцию.»
- «Все тесты зелёные, можно мержить.»
- «Этот тест упал — логика сломалась.»

---

### Линтер / ESLint
**Что это.** Автоматический «корректор стиля и ошибок» в коде. Линтер находит
подозрительные места (неиспользуемая переменная, опечатка в имени) и заставляет
держать код опрятным. ESLint — наш линтер.

**3 примера на практике:**
- ESLint подсвечивает забытую переменную как предупреждение.
- Серьёзные ошибки (необъявленное имя) линтер помечает как ошибку.
- `npm run lint:fix` сам исправляет часть замечаний.

**3 примера в предложении:**
- «Прогони линтер перед PR.»
- «ESLint ругается на неиспользуемый импорт.»
- «Поправь замечания линтера.»

---

### Сборка (build)
**Что это.** Превращение исходного кода в готовый к публикации пакет файлов,
который понимает браузер. Как из рукописи сделать напечатанную книгу. Если сборка
падает — приложение не выкатить.

**3 примера на практике:**
- `npm run build` собирает приложение в папку `dist/`.
- Перед PR проверяем, что сборка проходит без ошибок.
- Vercel при деплое сам запускает сборку.

**3 примера в предложении:**
- «Сборка прошла, ошибок нет.»
- «Этот коммит сломал сборку.»
- «Проверь `npm run build` перед PR.»

---

### npm
**Что это.** Менеджер пакетов и команд для проекта. Через него ставят чужие
библиотеки (`npm install`) и запускают команды проекта (`npm run dev`,
`npm test`). Думай о нём как о «пульте управления» проектом.

**3 примера на практике:**
- `npm install` ставит все нужные библиотеки.
- `npm run dev` запускает приложение для разработки.
- `npm test` прогоняет тесты.

**3 примера в предложении:**
- «Сделай `npm install` после клонирования.»
- «Запусти `npm run dev`.»
- «Все npm-команды описаны в `package.json`.»

---

### Зависимость / библиотека (dependency, library)
**Что это.** Чужой готовый код, который мы подключаем, чтобы не писать всё с нуля.
Например, иконки (lucide-react) или клиент Supabase. «Зависимость» — потому что
наш проект зависит от неё в работе.

**3 примера на практике:**
- `@supabase/supabase-js` — библиотека для работы с базой.
- `lucide-react` — библиотека иконок.
- Все зависимости перечислены в `package.json` и ставятся `npm install`.

**3 примера в предложении:**
- «Добавь библиотеку иконок в зависимости.»
- «Эта зависимость уже стоит.»
- «Обнови зависимость до свежей версии.»

## 6. Общие понятия программирования

### API
**Что это.** «Окошко выдачи» для общения программ между собой: набор правил, как
запросить данные или действие. Наш фронтенд через API-слой (`src/lib/api.js`)
просит у базы данные, не зная её внутренней кухни. Как окно в банке: говоришь
«хочу выписку» — получаешь, не заходя в хранилище.

**3 примера на практике:**
- Экран фондов берёт данные не из базы напрямую, а через API-слой.
- API-слой прячет детали Supabase, чтобы компоненты были проще.
- Когда модуль переводят с моков на Supabase — его подключают к API.

**3 примера в предложении:**
- «Этот модуль уже ходит через API-слой.»
- «Добавь метод в `api.js` для нового запроса.»
- «Фронтенд общается с базой только через API.»

---

### Моки (mock data, «заглушки»)
**Что это.** Временные выдуманные данные «для вида», пока не подключена настоящая
база. Позволяют рисовать и проверять интерфейс до того, как готов бэкенд. Как
манекен в витрине вместо живого человека.

**3 примера на практике:**
- Модули `crm`, `dashboard`, `stats` пока работают на моках в `src/data/`.
- Моки дают увидеть, как выглядит экран со списком, до подключения базы.
- По мере готовности модуль переводят с моков на реальный API.

**3 примера в предложении:**
- «Эти данные пока моковые.»
- «Замени моки на реальный запрос к Supabase.»
- «Моки лежат в `src/data/`, не в компонентах.»

---

### Транзакция / атомарность (transaction, atomic)
**Что это.** Несколько действий, которые должны выполниться **все вместе или
никак**. Если на середине что-то сломалось — откатывается всё, чтобы данные не
остались «наполовину». Как перевод денег: либо и списалось, и зачислилось, либо
ничего.

**3 примера на практике:**
- Запись дохода и обновление баланса фонда происходят в одной транзакции.
- Если списание не прошло — вся операция откатывается, баланс не «провисает».
- RPC-функции базы выполняют сложную логику атомарно.

**3 примера в предложении:**
- «Оберни эти шаги в одну транзакцию.»
- «Операция атомарна: всё или ничего.»
- «Транзакция откатилась из-за ошибки.»

---

### Идемпотентность (idempotency)
**Что это.** Свойство «повторный запуск ничего не ломает». Если по ошибке нажать
кнопку дважды или повторить запрос — результат будет такой же, как от одного раза,
а не два списания. Защита от дублей.

**3 примера на практике:**
- Повторная отправка одной и той же операции дохода не создаёт второй записи.
- Если сеть оборвалась и запрос повторили — деньги не спишутся дважды.
- Реестр спроектирован идемпотентно, чтобы не плодить дубли операций.

**3 примера в предложении:**
- «Сделай операцию идемпотентной.»
- «Повтор запроса безопасен — идемпотентность есть.»
- «Без идемпотентности будут двойные списания.»

---

### Рефакторинг (refactoring)
**Что это.** Улучшение структуры кода **без изменения того, что он делает**.
Как навести порядок в комнате: вещи те же, но всё аккуратно и легче найти. В
проекте действует «правило бойскаута»: тронул код — оставь его чуть чище.

**3 примера на практике:**
- Вынести повторяющуюся логику в отдельную функцию — рефакторинг.
- Перевести затронутый `.js`-файл на TypeScript по ходу правки.
- Разбить огромный компонент на несколько маленьких.

**3 примера в предложении:**
- «Это рефакторинг, поведение не меняется.»
- «Заодно отрефактори этот кусок.»
- «Большой рефакторинг сейчас не делаем — только по мере касания.»

---

### Баг / фикс (bug / fix)
**Что это.** **Баг** — ошибка в программе, из-за которой что-то работает не так.
**Фикс** — исправление этой ошибки. Слово «баг» (жучок) прижилось исторически.

**3 примера на практике:**
- Баг: баланс показывается без копеек — фикс в функции форматирования.
- Баг: на телефоне таблица вылезает за экран — фикс в адаптивной вёрстке.
- После фикса добавляем тест, чтобы баг не вернулся.

**3 примера в предложении:**
- «Нашёл баг в расчёте — сейчас зафикшу.»
- «Этот фикс закрывает ошибку из PR.»
- «Баг воспроизводится только на мобильном.»

---

### Логи (logs)
**Что это.** «Журнал событий» программы: записи о том, что и когда происходило,
особенно ошибки. Когда что-то сломалось, первым делом смотрят логи — как чёрный
ящик самолёта.

**3 примера на практике:**
- Если деплой упал — читаем логи сборки на Vercel.
- Логи базы (Supabase) помогают понять, почему запрос вернул ошибку.
- В логах видно точное сообщение об ошибке и где она возникла.

**3 примера в предложении:**
- «Посмотри логи деплоя.»
- «В логах есть ошибка подключения к базе.»
- «Логи подскажут, что именно сломалось.»

---

## 7. Инструменты автоматизации (MCP, CLI, боты)

### MCP (Model Context Protocol) / коннектор
**Что это.** Способ дать мне (Claude) «руки» — подключить внешние инструменты:
GitHub, Supabase, Vercel. Через MCP-коннектор я могу не просто советовать, а
реально применять миграции, открывать PR, смотреть деплои. Это как набор
подключённых инструментов на верстаке.

**3 примера на практике:**
- Через Supabase MCP я применяю миграции (если коннектор включён в окружении).
- Через GitHub MCP я открываю PR, читаю комментарии, смотрю CI.
- Если коннектор не включён — нужных инструментов у меня нет, и я об этом скажу.

**3 примера в предложении:**
- «Включи Supabase-коннектор (MCP), чтобы я применил миграцию.»
- «Через MCP я создам PR.»
- «Этот инструмент доступен только при подключённом MCP.»

---

### CLI (командная строка)
**Что это.** Текстовый способ управлять программами — командами, набранными в
терминале, вместо кнопок. `npm run dev`, `git push` — это всё команды CLI.
Быстрее и точнее, чем мышкой, когда привыкнешь.

**3 примера на практике:**
- `git status` в CLI показывает, что изменилось.
- `npm run build` запускает сборку из командной строки.
- Многие инструменты управляются только через CLI.

**3 примера в предложении:**
- «Выполни эту команду в CLI.»
- «В командной строке набери `npm test`.»
- «У этого инструмента есть CLI.»

---

### CI / CD (непрерывная интеграция и доставка)
**Что это.** Автоматические проверки и выкатки. **CI** — при каждом изменении сами
запускаются тесты/сборка, чтобы поймать поломку рано. **CD** — одобренное
изменение само уезжает в продакшен. Конвейер, который проверяет и публикует код
без ручной возни.

**3 примера на практике:**
- При открытии PR CI прогоняет проверки — если красные, не мержим.
- После мержа CD автоматически выкатывает версию на Vercel.
- CI ловит сломанную сборку до того, как она попадёт на прод.

**3 примера в предложении:**
- «CI на этом PR красный — смотрим, что упало.»
- «CD сам выкатит после мержа.»
- «Дождёмся зелёного CI и смержим.»

---

### Бот (bot)
**Что это.** Программа, которая автоматически выполняет рутину и пишет
комментарии. Например, Vercel-бот сам создаёт превью под PR и присылает ссылку.
Не человек, а помощник-автомат.

**3 примера на практике:**
- Vercel-бот публикует превью-ссылку в комментарии под PR.
- Бот проверок отмечает, прошёл ли CI.
- Боты экономят время на повторяющихся действиях.

**3 примера в предложении:**
- «Ссылку взял из комментария Vercel-бота.»
- «Бот ещё не успел собрать превью.»
- «Этот комментарий оставил бот, а не человек.»
