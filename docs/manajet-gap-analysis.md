# Gap-анализ ManaJet → «Яккасарой Финанс»

Дата ревизии: 28.06.2026  
Ветка: `claude/codex-integration-fbcg3z`  
Режим работы с ManaJet: read-only, без создания/изменения/удаления данных.

## 1. Резюме

Текущая реализация «Яккасарой Финанс» уже закрывает основу модели ManaJet по финансовому ядру: периоды ФП чт–ср, Директива с тремя этапами ФРС, доходы со счётом ДС/способом оплаты/контрагентом, фонды, ЗРС-заявки от поста, счета поставщиков и клиентов, частичные оплаты, Реестр `fp_register`, аудит, статистики с квотами, оргсхема, шляпы, CRM-воронка, задачи, уведомления, рассылки и read-only зеркало ManaJet.

Главные пробелы теперь не в базовом ФП, а в слоях, которые ManaJet держит вокруг него:

- **Бухгалтерский леджер и ОСВ**: ManaJet имеет `AccountingReg`, `AccountingStock`, `SheetOfAccounts`, `register_element` с дебетом/кредитом, НДС и проводками. У нас `fp_register` силён как ФП-леджер, но не является полноценным бухгалтерским контуром.
- **Маршрутные формы и ЗРС-коммуникации**: ManaJet отделяет `RoutingForm`/шаблоны/шаги/возврат на шаг назад и `MsgCsw` как канал пост→пост. У нас ЗРС есть в заявках, но нет универсального движка маршрутов и коммуникационного центра.
- **Workflow как универсальный поток**: CRM-воронка реализована, но ManaJet-модель `WorkflowBoard/List/Card/Step` шире: доски, карточки, шаги, шаблоны, статистики, фильтры по срокам и привязка к разным процессам.
- **Custom fields, генерация документов, вложения-шаблоны**: ManaJet широко использует `customMasterFields`, `DocumentAttachmentGenerator`, `*Attachment/ExportFromTemplate`. У нас есть вложения в ключевых местах, но нет универсального конструктора полей и генератора документов.
- **Права и настройки уровня платформы**: у нас есть RLS и роли, но ManaJet UI показывает настройки компании, API-ключи, week_begin_day, языки, интеграции, историю событий с типами. Не хватает отдельного админ-контура для управления этими слоями.

P0 для ближайшего исправления: не ломать уже рабочий ФП, а закрывать цепочку end-to-end там, где она ещё рвётся для управленца: ОСВ/двойная запись как read-only отчёт поверх Реестра, универсальные сообщения/комментарии пост→пост, расширение задач до статуса «выполнена → одобрена», и связка Workflow/CRM/задачи/статистики в едином процессе.

## 4. Детальный список пробелов

### 4.1 Финансовое планирование и Директива

#### P1 · Протокол Директивы как документ

- **Отсутствует:** экран/кнопка генерации протокола ФП, печать/экспорт Директивы.
- **ManaJet:** UI `Finance/Home?mode=Directive` показывает закрытие периода, три этапа распределения и встроенный список заявок. В API есть `DocumentAttachmentGenerator`, `BillAttachment/ExportFromTemplate`, `InvoiceAttachment/ExportFromTemplate`, что задаёт общий паттерн генерации документов.
- **Почему важно:** финсовет должен получить фиксируемое распоряжение для бухгалтерии и последующего аудита.
- **Объём:** M.
- **Зависимости:** шаблон документа, Storage, таблица/поле для ссылки на протокол в `directives` или `fp_periods`.
- **Приоритет:** P1.

#### P1 · Расчёт точки безубыточности

- **Отсутствует:** экран/виджет расчёта необходимой выручки снизу вверх по ФРС.
- **ManaJet/ХМС-логика:** `docs/manajet-логика.md` §4 описывает расчёт необходимого СКД → маржи → выручки.
- **Почему важно:** без этого Директива распределяет факт, но не помогает финансовому директору планировать нужный доход.
- **Объём:** M.
- **Зависимости:** фонды, правила распределения, планируемые постоянные суммы.
- **Приоритет:** P1.

#### P1 · Полный сценарий «заседание ФП»

- **Отсутствует:** пошаговый чек-лист процедуры ФП: сверка ДС → план дохода → распределение → долги → реклама → экономия → ЗП → будущие расходы.
- **ManaJet:** UI Директивы уже объединяет распределение и заявки; методология требует 15 действий.
- **Почему важно:** система сейчас проводит финоперации, но не ведёт финсовет по полной процедуре.
- **Объём:** L.
- **Зависимости:** отчёты, задачи, протокол Директивы.
- **Приоритет:** P1.

### 4.2 Реестр, отчёты, бухгалтерский контур

#### P0 · Двойная запись дебет/кредит и ОСВ

- **Отсутствует:** бухгалтерский слой с планом счетов, дебетом/кредитом, ОСВ.
- **ManaJet:** Swagger: `Register.register_element`, `AccountingReg`, `AccountingStock`, `SheetOfAccounts`; UI «Расходы/Фонды/Контроль средств» показывает таблицы операций с колонками `Дата оплаты`, `Источник`, `Назначение`, `Тип операции`, `Информация`, `Сумма`.
- **У нас:** `fp_register` хранит финансовое движение ФП со знаком, `op_type`, `fund_id`, `cash_account_id`, `counterparty_id`, `payment_type_id`, но не пару счетов дебет/кредит.
- **Почему важно:** без ОСВ бухгалтер/ФД не сможет сверять ManaJet-подобные проводки и строить отчёты по плану счетов.
- **Объём:** L.
- **Зависимости:** таблицы `chart_of_accounts`/`accounting_register`, миграция, pgTAP, read-only маппинг из `fp_register`, затем UI отчёта ОСВ.
- **Приоритет:** P0.

#### P0 · Бухгалтерские проводки по НДС и складу

- **Отсутствует:** `AccountingStock`, `StockDocumentVat`, НДС в счетах и складских проводках.
- **ManaJet:** Swagger: `AccountingStock`, `ApiBillOutput.total_without_amount`, `total_vat_amount`, `is_vat_enabled`, `vat_calculation_type`, `detail[]`.
- **У нас:** поставщик/клиентские счета без НДС и табличной части.
- **Почему важно:** для управленческого ФП можно жить без НДС, но для ManaJet-паритета и бухсверки это крупный разрыв.
- **Объём:** L.
- **Зависимости:** решение: делать внутри системы или отдать iiko/бухгалтерии.
- **Приоритет:** P0 для gap-паритета, P2 если склад/НДС остаются во внешней системе.

#### P1 · История событий с типами ManaJet

- **Отсутствует:** фильтры истории по событию/типу: Системные, Воронка продаж, Пользовательские; Персонал, Компания, Структура, Бизнес-процесс, Финансовое Планирование, Настройка аккаунта.
- **ManaJet:** UI `History` показывает эти фильтры. API: `History`, `HistoryEvent`.
- **У нас:** `audit_log` и экран журнала аудита есть, но типизация событий проще.
- **Почему важно:** аудит сейчас технический; ManaJet делает его управленческим.
- **Объём:** M.
- **Зависимости:** расширение `audit_log` типами, миграция, UI-фильтры.
- **Приоритет:** P1.

### 4.3 Заявки, ЗРС, RoutingForm и коммуникации

#### P0 · Универсальный RoutingForm

- **Отсутствует:** отдельный модуль маршрутных форм, шаблонов, шагов, sequence, custom fields.
- **ManaJet:** UI `RoutingForm/RoutingForm`: «Бизнес-процессы», «Шаблоны бизнес-процессов», «Начать бизнес-процесс», вкладки «Входящие», «Текущие», «Просроченные», «Завершённые», «Запрос на шаг назад». Swagger: `RoutingForm`, `RoutingFormTemplate`, `ApiRoutingFormStep`, `RoutingFormStepPostAPI`.
- **У нас:** ЗРС реализована как поля заявки `csw_data/csw_situation/csw_solution`, статусы и комментарии, но маршрут фиксирован вокруг финкомитета.
- **Почему важно:** ManaJet использует маршруты не только для расходов: договоры, найм, новая точка, согласования.
- **Объём:** L.
- **Зависимости:** новые таблицы `routing_form_templates`, `routing_forms`, `routing_form_steps`, `routing_form_fields`; RLS по постам; уведомления.
- **Приоритет:** P0.

#### P0 · ЗРС-сообщения пост→пост

- **Отсутствует:** коммуникационный центр `MsgCsw`: кому/от кого по посту, черновики, входящие/исходящие, метки.
- **ManaJet:** UI `Msg` показывает «Коммуникации», входящие/отложенные/исходящие/черновики, «Написать сообщение». Swagger: `Msg`, `MsgCsw`, `PositionsToCSW`, `PositionsFromCSW`.
- **У нас:** комментарии по заявкам/задачам и уведомления, но нет универсального сообщения от поста к посту.
- **Почему важно:** ХМС-заявка и коммуникация должны идти по оргсхеме, а не только по пользователям.
- **Объём:** L.
- **Зависимости:** `messages`, `message_recipients`, `position_id_from/to`, RLS `holds_position`, связь с задачами/заявками.
- **Приоритет:** P0.

#### P1 · SLA и возврат на шаг

- **Отсутствует:** `dt_term`, `speed_ratio`, `requested_return_status`, `request_description` на уровне маршрута.
- **ManaJet:** Swagger `ApiRoutingFormStep` содержит срок, скорость согласования, запрос возврата.
- **У нас:** есть `revision` для заявки, но нет SLA и возврата на конкретный маршрутный шаг.
- **Почему важно:** финкомитет видит статус, но не управляет временем согласования.
- **Объём:** M.
- **Зависимости:** RoutingForm.
- **Приоритет:** P1.

### 4.4 Счета поставщиков и клиентов

#### P1 · НДС и табличная часть счёта

- **Отсутствует:** позиции счёта, сумма без НДС, НДС, итог, тип расчёта НДС.
- **ManaJet:** `ApiBillOutput.detail[]`, `total_without_amount`, `total_vat_amount`, `is_vat_enabled`, `vat_calculation_type`.
- **У нас:** счёт хранит агрегированную сумму и вложения.
- **Почему важно:** для поставщиков и договоров нужна детализация и печатные формы.
- **Объём:** L.
- **Зависимости:** таблица `bill_lines`/`invoice_lines`, миграция, UI-табличная часть.
- **Приоритет:** P1.

#### P1 · Генерация документов из шаблонов

- **Отсутствует:** единый механизм шаблонов для счёта/договора/акта/шляпы.
- **ManaJet:** `BillAttachment/ExportFromTemplate`, `InvoiceAttachment/ExportFromTemplate`, `DocumentAttachmentGenerator`.
- **У нас:** есть локальная печать клиентского счёта и Storage-вложения, но нет шаблонизатора.
- **Почему важно:** снижает ручную работу с договорами банкетов и счетами.
- **Объём:** L.
- **Зависимости:** шаблоны DOCX/PDF, Storage, очередь генерации.
- **Приоритет:** P1.

#### P2 · Возврат поставщику как отдельный сценарий

- **Отсутствует:** UI-сценарий `bill_return`.
- **ManaJet:** `BillPaymentInput.bill_return`.
- **У нас:** есть откат оплаты счёта, но бизнес-сценарий возврата поставщику не выделен.

## 5. Расхождения бизнес-логики ХМС

### Фонды и этапы ФРС

- ManaJet UI на Директиве показывает один и тот же фонд `FD6 - Фонд Резервов` в разных этапах, а в методологии этап является свойством правила распределения.
- У нас после `docs/funds-spec.md` принято решение: фонд имеет один `funds.stage`, смена этапа синхронизируется через `fp_set_fund_stage`, прежнее «ФД6 с двух этапов» отменено.
- Это осознанное расхождение с ранним ТЗ v2 и частичное расхождение с видимым ManaJet UI. Нужно зафиксировать у заказчика: ManaJet-паритет важнее или текущая упрощённая модель фондов важнее.

### Реестр

- ManaJet `Register` — двусторонний леджер через `register_element` и связанные счета.
- У нас `fp_register` — источник истины ФП, неизменяемый, со сторно и триггерами балансов, но без полноценного дебет/кредит.
- Риск: при интеграции с бухгалтерией появится второй источник интерпретации денег. Рекомендация: не менять `fp_register`, а добавить read-only accounting projection.

### Периоды ФП

- Совпадение: чт–ср, `FpPlan.is_executive_confirmed` и `is_baf_confirmed` уже отражены в `fp_periods`.
- Расхождение: ManaJet настройки `week_begin_day` есть в Setup; у нас правило периода фактически продуктово зафиксировано.
- Рекомендация: не делать настройку периода до явного требования, но хранить в будущих `company_settings`.

### ЗРС-заявки

- У нас ЗРС-поля в заявке уже есть.
- ManaJet шире: ЗРС — ещё и сообщение/маршрут по постам (`MsgCsw`, `RoutingForm`).
- Риск: пользователь будет ожидать «ЗРС» во всех коммуникациях, а не только в расходах.

### Права RLS

- У нас права сильные: роли, точки, фонды, статьи, `holds_position`.
- ManaJet UI показывает ещё группы сотрудников, личные/подчинённые фильтры, настройки API и интеграций.
- Пробел не в RLS-основе, а в админ-UI и тонкой матрице «пост/группа/подчинённые/шаблон процесса».

## 6. End-to-end сценарии

### Доход → распределение → Реестр

Статус: в основном работает.

1. Доход вводится с видом, счётом ДС, способом оплаты, контрагентом и документом.
2. Триггер/RPC создаёт запись Реестра.
3. Директива распределяет по этапам.
4. Фонды получают баланс.

Где рвётся относительно ManaJet:

- нет планирования дохода/заделов как части процедуры ФП;
- нет бухгалтерского debet/kredit представления операции;
- нет автоматического iiko/банк импорта.

### Заявка/счёт → одобрение → оплата → Реестр

Статус: рабочий ФП-контур есть.

1. Заявка создаётся от поста, в ЗРС-формате.
2. Одобрение может отличаться от запрошенной суммы.
3. Оплата частичная/полная, с периодом оплаты.
4. Сторно/откат есть.

Где рвётся относительно ManaJet:

- нет универсального маршрута согласования;
- нет SLA/скорости шага;
- нет ЗРС-сообщений пост→пост;
- нет проводок ОСВ.

### CRM → счёт клиента → доход → Реестр

Статус: частично рабочий.

1. Лид/клиент/счёт существуют.
2. Оплата счёта создаёт доход.
3. Вложения и откат оплаты есть.

Где рвётся:

- договор/счёт не генерируется из шаблона ManaJet-типа;
- нет графика оплат и автоматических напоминаний клиенту;
- Massmail пока ручной через WhatsApp-ссылки.

### Статистика → состояние → ЗП

Статус: базово есть.

1. Статистики и квоты есть.
2. Состояния и коэффициенты ЗП есть.
3. ЗП рассчитывается по баллам.

Где рвётся:

- не все `stat_type` расшифрованы;
- нет формульных/автоматических статистик;
- состояние поста не всегда автоматически связано с расчётом ЗП.

## 10. Явные баги/несоответствия, найденные по ходу

1. В `src/data/navigation.js` есть модуль `projects`, но в `MODULE_NAV` и рендере продукта он не имеет полноценной навигации. Это либо будущий модуль, либо пункт меню-заглушка. Для ManaJet-паритета это связано с `Task/AdminScale` и Workflow.
2. Раннее ТЗ v2 говорит «этап — свойство правила, не фонда; ФД6 может пополняться с двух этапов», а `docs/funds-spec.md` и текущая реализация закрепили один этап на фонд. Это не баг кода, но важное расхождение требований.
3. В Swagger `ApiBillOutput` содержит `id_paymend_metod` с опечаткой ManaJet. При интеграции нельзя автоматически «исправлять» имя на `payment_method` без маппера.
4. ManaJet UI в финансовых таблицах остаётся табличным и широким; у нас по правилам проекта мобильные списки должны быть карточками. При реализации недостающих экранов не копировать таблицы ManaJet один-в-один.
