# ADR-0007. Миграции: сначала staging, потом прод

- **Дата**: 2026-07-01 · **Статус**: принято (заказчик подтвердил оплату ветки)
- **Решение**: staging-ветка Supabase `qa-staging` (project_ref `xviacaxhfucinvminbtp`,
  ~$10/мес) — обязательная первая остановка каждой миграции: при дрейфе `rebase_branch`,
  затем `apply_migration` на ветку, поведенческая проверка/pgTAP, и только потом прод.
- **Почему**: до этого миграции применялись сразу в единственный прод с реальными
  деньгами; ошибка схемы была бы необратимой. Данных прода в ветке нет — для
  поведенческих тестов на фикстурах этого достаточно.
- **Что меняет**: раздел «База данных» CLAUDE.md, runbook в `supabase/README.md`;
  первая миграция по процессу — `20260702090000_fp_postings_offperiod_date.sql`.
