# ADR-0009. Мониторинг ошибок фронта: журнал в Supabase + опциональный Sentry

- **Дата**: 2026-07-02 · **Статус**: принято
- **Решение**: два независимых канала мониторинга. Основной — таблица
  `client_errors` в Supabase: `src/lib/monitoring.ts` best-effort пишет туда ошибки
  рендера (ErrorBoundary), необработанные исключения и promise-реджекты вошедших
  пользователей (лимит 20/сессию против лавины); просмотр и архив — финадминам в
  «Журнал аудита» → «Ошибки фронта». Дополнительный — Sentry, включается переменной
  `VITE_SENTRY_DSN` в Vercel (регистрация — за заказчиком); без DSN SDK не грузится.
- **Почему**: журнал в БД работает сразу, без внешних аккаунтов, и покрывает главный
  сценарий («у управляющего на телефоне ошибка — теперь видна владельцу»). Sentry
  даёт группировку/алерты, но требует действий заказчика — оставлен опцией.
- **Что меняет**: миграция `20260702120000_client_errors.sql` (RLS: insert — свой
  authenticated, select/update — is_fin_admin); `src/lib/api/errors.ts`; вкладка в
  `AuditLog.jsx`. Ошибки на экране входа (нет сессии) в БД не пишутся — их ловит
  только Sentry, если подключён.
