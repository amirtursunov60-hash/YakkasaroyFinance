# Глоссарий технических терминов проекта Yakkasaroy

Это словарь **технических слов**, которые я (Claude) использую, когда мы работаем над вашим приложением. Цель — чтобы вы понимали не просто «что это за слово», а **зачем оно существует и как устроено по сути**.

Каждый термин разобран по схеме:
- **Простыми словами** — что это, человеческим языком.
- **Суть (зачем это нужно)** — концептуальное понимание, аналогия из жизни.
- **3 примера употребления** — как это слово реально звучит в нашей переписке.

> Это про **технические** термины (репозиторий, Supabase, ветка и т.д.). Бизнес-термины ХМС (ФП, ФРС, фонд, Директива, ЗРС, шляпа, Реестр…) — это отдельный словарь, он лежит в ТЗ, Приложение А.

## 1. Хранение и версии кода

### Репозиторий (repository, «репо»)
- **Простыми словами:** папка с вашим проектом, у которой есть «память» — она хранит весь код **и всю историю** его изменений с самого начала.
- **Суть:** обычная папка помнит только последнюю версию файла. Репозиторий помнит **каждую** версию: кто, когда и что менял. Это как бухгалтерская книга для кода — можно отмотать назад к любому дню и увидеть, как было. У вашего проекта два репозитория: `YakkasaroyFinance` (приложение-финансы) и `yakkasaroy-menu` (меню).
- **Примеры:**
  1. «Я закоммитил изменения в репозиторий.»
  2. «Этот файл лежит в корне репозитория.»
  3. «У нас два репозитория — финансы и меню, они независимы.»

### Git
- **Простыми словами:** программа, которая ведёт эту «память» репозитория. Сам механизм версий.
- **Суть:** Git — это «машина времени» для кода. Репозиторий — это результат её работы (хранилище), а Git — инструмент, который записывает историю и умеет переключаться между версиями, сравнивать их, объединять. GitHub — это сайт, где эти репозитории живут в облаке; Git — технология под капотом.
- **Примеры:**
  1. «Git показывает, что в файле изменились 3 строки.»
  2. «История Git хранит все версии Директивы-экрана.»
  3. «Через Git я откатил неудачное изменение.»

### GitHub
- **Простыми словами:** сайт-облако, где хранятся репозитории, чтобы к ним был доступ из любого места и можно было работать совместно.
- **Суть:** Git живёт у каждого на компьютере. Чтобы код был в одном общем месте (и не пропал, если сломается ноутбук), его выкладывают на GitHub. Это «облачный дом» для кода + площадка для обсуждений, ревью и автоматизации. Ваши репозитории живут именно на GitHub.
- **Примеры:**
  1. «Я запушил ветку на GitHub.»
  2. «Pull Request создаётся на GitHub.»
  3. «Код доступен в репозитории на GitHub.»

### Ветка (branch)
- **Простыми словами:** отдельная «дорожка» для работы, копия проекта, где можно спокойно вносить изменения, не трогая основную рабочую версию.
- **Суть:** представьте, что вы редактируете важный документ. Вместо того чтобы править оригинал, вы делаете копию, экспериментируете в ней, и только когда всё готово и проверено — переносите в оригинал. Ветка — это такая копия-дорожка. Основная дорожка называется `main`. Сейчас мы работаем в ветке `claude/tech-terms-glossary-w9qimm` — отдельной, чтобы не задеть рабочее приложение.
- **Примеры:**
  1. «Я создал новую ветку для этой задачи.»
  2. «Все изменения делаю в ветке, а не сразу в main.»
  3. «Эта ветка готова к слиянию.»

### Коммит (commit)
- **Простыми словами:** одно сохранение-снимок изменений с подписью, что именно поменяли.
- **Суть:** коммит — это как «сохранить» в игре, но с комментарием. Каждый коммит фиксирует пачку изменений и описывает их человеческим текстом («добавил словарь терминов»). Из таких снимков складывается вся история проекта, и к любому из них можно вернуться.
- **Примеры:**
  1. «Я сделал коммит с понятным описанием.»
  2. «В этом коммите — только правки экрана Фонды.»
  3. «Откатимся на коммит, где всё ещё работало.»

### Push / Pull / Fetch (запушить / подтянуть / получить)
- **Простыми словами:** **push** — отправить свои сохранения в облако (GitHub). **Pull** — забрать оттуда свежие изменения себе. **Fetch** — посмотреть, что появилось в облаке, не забирая пока.
- **Суть:** ваш компьютер и GitHub — два хранилища одного кода. Они не синхронизируются сами. Push — «выгрузить наверх», Pull — «загрузить вниз к себе». Это как отправить и получить посылку между вашим столом и общим складом.
- **Примеры:**
  1. «Я запушил коммит — теперь он в облаке.»
  2. «Сделай pull, чтобы получить мои правки.»
  3. «Fetch показал, что появилась новая ветка.»

### Pull Request (PR, «пиар», запрос на слияние)
- **Простыми словами:** официальная заявка: «я закончил в своей ветке, прошу проверить и влить эти изменения в основную версию».
- **Суть:** PR — это «дверь» между вашей рабочей дорожкой и основной. Перед тем как изменения попадут в боевое приложение, они оформляются в PR: там их видно списком, можно обсудить, а робот Vercel автоматически собирает превью-версию для проверки. В вашем проекте правило: **каждое изменение — отдельный PR в `main`**, и по одному за раз.
- **Примеры:**
  1. «Я открыл Pull Request с новым экраном.»
  2. «По PR Vercel создаёт превью-ссылку.»
  3. «Дождёмся, пока вы смержите PR, и возьмём следующее.»

### Merge (мёрж, слияние)
- **Простыми словами:** объединить изменения из вашей ветки в основную — «влить» работу в боевую версию.
- **Суть:** пока изменения в отдельной ветке — они никого не затрагивают. Merge — момент, когда они становятся частью основного проекта. После мёржа PR в `main` обновляется боевое приложение.
- **Примеры:**
  1. «После мёржа в main изменения уйдут в продакшен.»
  2. «Git сам объединил две ветки при мёрже.»
  3. «Не берусь за новое, пока вы не смержите текущий PR.»

### Main (главная ветка)
- **Простыми словами:** основная, «чистовая» дорожка проекта — то, что считается официальной рабочей версией.
- **Суть:** среди всех веток одна главная — `main`. Её берегут: напрямую в неё не пишут, всё попадает туда только через проверенный PR. Боевой сайт на Vercel показывает именно то, что лежит в `main`.
- **Примеры:**
  1. «Напрямую в main не пушим — только через PR.»
  2. «Продакшен обновляется из ветки main.»
  3. «Эта ветка отстала от main, надо подтянуть.»

## 2. Публикация приложения

### Деплой (deploy, развёртывание)
- **Простыми словами:** процесс, когда код превращается в работающий сайт, доступный по ссылке в интернете.
- **Суть:** код сам по себе — это просто текст в файлах. Деплой — это «публикация»: взять код, собрать его и разместить на сервере, чтобы любой мог открыть приложение в браузере. Как сдать рукопись в типографию и получить готовую книгу в магазине.
- **Примеры:**
  1. «После мёржа произойдёт деплой на боевой адрес.»
  2. «Деплой превью готов — вот ссылка.»
  3. «Деплой упал из-за ошибки в сборке.»

### Vercel (Версель)
- **Простыми словами:** сервис, который автоматически публикует ваше приложение в интернете.
- **Суть:** Vercel «следит» за вашим репозиторием. Появился PR — он сам собирает превью-версию. Смержили в `main` — он сам обновляет боевой сайт. Вам не нужно ничего вручную загружать на сервер; Vercel делает деплой автоматически.
- **Примеры:**
  1. «Vercel-бот прислал превью-ссылку под PR.»
  2. «Переменные окружения задаются в панели Vercel.»
  3. «Боевой сайт хостится на Vercel.»

### Preview (превью-деплой, превью-ссылка)
- **Простыми словами:** временная рабочая версия приложения с изменениями из конкретного PR — чтобы посмотреть и проверить до того, как влить в боевую.
- **Суть:** превью — это «черновик вживую». Те же изменения, но на отдельном временном адресе. Можно открыть, потыкать, показать заказчику — и только потом решать, мёржить ли. Именно короткую превью-ссылку (`https://yakkasaroy-finance-...vercel.app/`) я присылаю вам в чат для проверки.
- **Примеры:**
  1. «Вот превью-ссылка — посмотрите новый экран.»
  2. «На превью видно, как выглядит на телефоне.»
  3. «Превью обновилось после нового коммита.»

### Продакшен (production, «прод», боевая версия)
- **Простыми словами:** настоящее, рабочее приложение, которым пользуются реально, — не черновик.
- **Суть:** есть «черновик» (превью) и есть «боевое» (продакшен). Продакшен — это та версия, что обновляется только после мёржа в `main`, на неё заходят настоящие пользователи. Поэтому туда попадает только проверенное.
- **Примеры:**
  1. «В продакшен уходит только то, что в main.»
  2. «На проде нельзя экспериментировать — там реальные данные.»
  3. «Изменение доедет до продакшена после мёржа.»

### Билд / Сборка (build)
- **Простыми словами:** процесс превращения исходного кода в готовый, оптимизированный набор файлов, который понимает браузер.
- **Суть:** код, который мы пишем, удобен человеку, но браузеру нужен другой, «упакованный» вид — сжатый и быстрый. Сборка делает этот перевод. Если в коде ошибка — сборка «падает» (не получается), и это сигнал, что что-то не так, ещё до публикации.
- **Примеры:**
  1. «Перед PR проверяю, что сборка проходит (`npm run build`).»
  2. «Сборка упала — есть опечатка в импорте.»
  3. «Готовая сборка лежит в папке `dist/`.»

## 3. База данных и сервер

### Supabase (Супабейс)
- **Простыми словами:** «облачный сервер всё-в-одном» для приложения: хранит данные, проверяет логины, хранит файлы. Готовый «движок» под капотом.
- **Суть:** приложению мало красивых экранов — ему нужно где-то **хранить данные** (фонды, доходы, сотрудников), **проверять, кто вошёл** и **что ему можно видеть**. Supabase даёт всё это сразу: базу данных, вход по логину, хранилище файлов, права доступа. Финансовый контур вашего проекта уже работает на реальном Supabase.
- **Примеры:**
  1. «Доходы сохраняются в Supabase.»
  2. «Вход в приложение проверяется через Supabase.»
  3. «Переводим оставшиеся модули с моков на Supabase.»

### База данных (БД, database)
- **Простыми словами:** организованное хранилище всех данных приложения — как огромная умная таблица-картотека.
- **Суть:** база данных — это «память» приложения. Не код, а сами данные: списки фондов, записи о доходах, сотрудники, банкеты. Устроена как набор таблиц, между которыми есть связи. Когда вы видите на экране сумму в фонде — она пришла из базы данных.
- **Примеры:**
  1. «Эти суммы лежат в базе данных, а не в коде.»
  2. «Запрос к базе вернул список сотрудников.»
  3. «В базе данных 33 таблицы по финансам.»

### PostgreSQL (Постгрес)
- **Простыми словами:** конкретная программа базы данных, которую использует Supabase. Сам «вид» базы.
- **Суть:** баз данных бывает много разных видов. Supabase внутри использует PostgreSQL — одну из самых надёжных и проверенных. Когда я говорю «Постгрес» — это про то, по каким правилам устроена и работает ваша база.
- **Примеры:**
  1. «Supabase построен на PostgreSQL.»
  2. «Это возможность PostgreSQL — триггеры на таблицах.»
  3. «Запрос написан на языке PostgreSQL (SQL).»

### Таблица (table)
- **Простыми словами:** один лист данных в базе — строки и колонки, как в Excel.
- **Суть:** база данных состоит из таблиц. Колонки — это какие поля у записи (например, у дохода: дата, сумма, точка, способ оплаты). Строки — сами записи. Таблица «фонды», таблица «доходы», таблица «сотрудники» — каждая про свой тип данных.
- **Примеры:**
  1. «Доход добавляется новой строкой в таблицу `incomes`.»
  2. «В таблице фондов есть колонка `balance`.»
  3. «Новая таблица заводится с правами доступа.»

### SQL (Эс-Кью-Эль, «скуль»)
- **Простыми словами:** язык, на котором «разговаривают» с базой данных — задают вопросы и команды.
- **Суть:** чтобы получить данные из базы или что-то изменить, нужно «попросить» на её языке. SQL — это и есть тот язык: «дай все доходы за эту неделю», «добавь такую-то строку». Почти все базы данных понимают SQL.
- **Примеры:**
  1. «SQL-запрос вернул сумму по фонду.»
  2. «Миграция — это, по сути, SQL-команды для базы.»
  3. «Заказчик SQL вручную не запускает — это делаю я.»

### Миграция (migration)
- **Простыми словами:** записанное изменение устройства базы данных (добавили таблицу/колонку/правило) — отдельный файл-«указание».
- **Суть:** структуру базы (какие таблицы, какие колонки, какие правила) нельзя менять как попало — иначе потеряется порядок и можно сломать данные. Поэтому каждое изменение оформляется **отдельным файлом-миграцией**: это как нотариально заверенная поправка к устройству базы. Старые миграции не правят — добавляют новые поверх.
- **Примеры:**
  1. «Добавление колонки оформил новой миграцией.»
  2. «Миграции применяю я через Supabase, не вы вручную.»
  3. «Сверяюсь со свежей миграцией перед изменением схемы.»

### Схема (schema)
- **Простыми словами:** «чертёж» базы данных — какие в ней есть таблицы, колонки, связи и правила.
- **Суть:** если данные — это содержимое, то схема — это **форма**, в которую они складываются. Какие таблицы существуют, какие у них поля, как они связаны. Изменить схему = изменить устройство хранилища (через миграцию).
- **Примеры:**
  1. «Изменение схемы сопровождается тестом.»
  2. «По схеме у дохода обязательно есть точка.»
  3. «Базовая схема — это весь финконтур по ТЗ.»

### RLS (Row Level Security, защита на уровне строк)
- **Простыми словами:** правила базы, которые решают, **какие именно строки** данных вправе видеть и менять конкретный человек.
- **Суть:** мало пустить человека в приложение — нужно, чтобы менеджер Худжанда видел свою точку, а не Душанбе; бухгалтер — своё, владелец — всё. RLS — это «вахтёр на уровне каждой строки»: база сама, в глубине, отсекает чужое. Это безопаснее, чем полагаться на экран, — данные защищены в самом хранилище.
- **Примеры:**
  1. «Права на точки держит RLS, а не интерфейс.»
  2. «Новая таблица заводится сразу с RLS.»
  3. «RLS не покажет менеджеру чужую точку.»

### Триггер (trigger)
- **Простыми словами:** автоматическое действие базы, которое срабатывает само в ответ на событие («если случилось X — сделай Y»).
- **Суть:** триггер — это «рефлекс» базы данных. Например: как только добавили доход — триггер сам пересчитывает баланс фонда и сам запрещает уйти в минус. Никто не нажимает кнопку — база делает это автоматически и надёжно. Поэтому балансы у вас никто не пишет руками: их ведут триггеры.
- **Примеры:**
  1. «Баланс фонда обновляет триггер, не клиент.»
  2. «Триггер запрещает операцию в закрытом периоде.»
  3. «Вставка дохода триггером создаёт запись в Реестре.»

### API (Эй-Пи-Ай, программный интерфейс)
- **Простыми словами:** «окошко выдачи» между приложением и базой/сервером — набор готовых способов попросить данные или отправить их.
- **Суть:** экран не лезет в базу напрямую — он обращается через API: «дай список доходов», «сохрани этот расход». API — это посредник-официант: вы говорите заказ, он приносит готовое, а кухню (базу) вы не трогаете. В вашем проекте этот слой-посредник — файл `src/lib/api.js`.
- **Примеры:**
  1. «Экран Фонды ходит в базу через API.»
  2. «Переводим модуль CRM на API-слой.»
  3. «API скрывает детали базы от интерфейса.»

### Аутентификация (Auth, вход/авторизация)
- **Простыми словами:** проверка, кто вы такой, при входе в приложение (логин-пароль), и определение, что вам можно.
- **Суть:** два связанных вопроса: «кто ты?» (аутентификация — вход по логину) и «что тебе можно?» (авторизация — права). Supabase Auth берёт это на себя: проверяет вход и выдаёт приложению понимание, кто сейчас работает, чтобы вместе с RLS показать только разрешённое.
- **Примеры:**
  1. «Вход в приложение — через Supabase Auth.»
  2. «После аутентификации подгружается профиль с ролью.»
  3. «Без входа приложение показывает экран логина.»

### Storage (хранилище файлов)
- **Простыми словами:** место в Supabase для файлов — картинок, сканов, документов, вложений.
- **Суть:** базы данных хороши для чисел и строк, но не для файлов. Для фото чека, скана договора, вложения к заявке есть отдельное хранилище — Storage. Данные — в базе, файлы — в Storage, и они ссылаются друг на друга.
- **Примеры:**
  1. «Скан счёта кладём в Storage.»
  2. «Вложения к заявке хранятся в Storage.»
  3. «Storage отдаёт ссылку на загруженный файл.»

### Бэкап (backup, резервная копия)
- **Простыми словами:** запасная копия данных на случай поломки или ошибки — чтобы можно было восстановить.
- **Суть:** данные бесценны, а сбои случаются. Бэкап — это «запасной ключ»: регулярная копия базы, из которой можно вернуть всё назад. Важно не только делать копии, но и проверять, что из них реально получается восстановиться (это называют restore-drill).
- **Примеры:**
  1. «Бэкап базы делается регулярно.»
  2. «Проверка восстановления из бэкапа — часть работы.»
  3. «Если что-то удалили — восстановим из бэкапа.»

## 4. Лицевая часть приложения (frontend)

### Frontend / Backend (фронтенд / бэкенд)
- **Простыми словами:** **Frontend** — всё, что вы видите и нажимаете (экраны, кнопки, формы). **Backend** — невидимая часть за кулисами (база, сервер, логика, права).
- **Суть:** приложение как ресторан: зал, меню и официант — это фронтенд (с этим общается гость); кухня, склад, бухгалтерия — это бэкенд (гость их не видит, но без них ничего не работает). У вас фронтенд — это React-экраны, бэкенд — Supabase.
- **Примеры:**
  1. «Это правка фронтенда — поменялся вид кнопки.»
  2. «Логику прав держит бэкенд (RLS в базе).»
  3. «Фронтенд просит данные у бэкенда через API.»

### React (Реакт)
- **Простыми словами:** инструмент (библиотека), на котором построены все экраны вашего приложения.
- **Суть:** React — это «конструктор интерфейсов». Он позволяет собирать экраны из переиспользуемых кусочков (компонентов) и сам обновляет картинку, когда данные меняются. Весь видимый интерфейс Yakkasaroy написан на React.
- **Примеры:**
  1. «Экран Директивы — это React-компонент.»
  2. «React сам перерисует список, когда придут данные.»
  3. «Новый экран добавляем как React-компонент.»

### Компонент (component)
- **Простыми словами:** самостоятельный кусочек интерфейса — кнопка, карточка, форма, целый экран. Кирпичик, из которых собран сайт.
- **Суть:** вместо одной гигантской страницы интерфейс собирают из деталей-компонентов, как из кубиков LEGO. Один раз сделал «карточку фонда» — используешь её много раз. Это удобно чинить и переиспользовать. У вас каждый экран и многие элементы — отдельные компоненты.
- **Примеры:**
  1. «Сделал компонент карточки фонда.»
  2. «Этот компонент переиспользуется на трёх экранах.»
  3. «Шапку вынес в отдельный компонент.»

### Состояние (state, «стейт»)
- **Простыми словами:** текущие данные, которые «помнит» экран прямо сейчас: что введено в форму, какая вкладка открыта, что выбрано.
- **Суть:** интерфейс живой — он меняется по ходу работы. State — это «оперативная память» экрана: открыт фильтр или нет, какая точка выбрана, что набрано в поле. Меняется state — React сам перерисовывает то, что видно.
- **Примеры:**
  1. «Выбранная неделя хранится в состоянии экрана.»
  2. «По клику меняем state — открывается модалка.»
  3. «Пока данные не в базе, держим их в state.»

### Vite (Вайт)
- **Простыми словами:** инструмент, который запускает приложение при разработке и собирает его для публикации. «Мотор» сборки.
- **Суть:** Vite делает две вещи: при разработке мгновенно показывает изменения в браузере (команда `npm run dev`), а для публикации — собирает оптимизированную версию (`npm run build`). Это рабочий станок, на котором крутится проект.
- **Примеры:**
  1. «`npm run dev` поднимает Vite — локальный сайт.»
  2. «Vite собирает финальную версию в `dist/`.»
  3. «Vite сразу показывает правку без перезагрузки.»

### Tailwind (Тейлвинд)
- **Простыми словами:** способ оформлять внешний вид (цвета, отступы, размеры) короткими готовыми пометками прямо на элементах.
- **Суть:** вместо того чтобы описывать стиль отдельно, Tailwind даёт набор готовых «наклеек» (например, отступ, цвет фона), которые вешаешь прямо на элемент. Быстро и единообразно. В вашем проекте Tailwind разрешён для новых экранов, цвета берутся из единой палитры темы.
- **Примеры:**
  1. «Новый экран оформил на Tailwind.»
  2. «Цвета в Tailwind берём из палитры темы.»
  3. «Отступы задал классами Tailwind.»

### Тема (theme) и тёмная/светлая тема
- **Простыми словами:** единый набор цветов и оформления приложения. Тёмная тема — светлый текст на тёмном фоне, светлая — наоборот.
- **Суть:** чтобы всё приложение выглядело цельно, цвета не пишут где попало, а собирают в одном месте — в «теме». Поменял там — поменялось везде. У вас по умолчанию тёмная тема в стиле Alif Business с зелёными акцентами; обе темы должны работать.
- **Примеры:**
  1. «Все цвета берём из палитры темы, не хардкодим.»
  2. «Проверил, что экран нормально смотрится в обеих темах.»
  3. «Зелёный для сумм — отдельный токен темы.»

## 5. Языки и инструменты разработки

### Код
- **Простыми словами:** текст-инструкция, написанная на языке программирования, по которой работает приложение.
- **Суть:** всё, что делает приложение, описано кодом — пошаговыми инструкциями для компьютера. Писать/менять код = менять поведение приложения. Код хранится в файлах внутри репозитория.
- **Примеры:**
  1. «Поправил код экрана — кнопка теперь работает иначе.»
  2. «Этот код отвечает за расчёт баланса.»
  3. «Код на русском в комментариях — так у нас принято.»

### JavaScript (JS) и JSX
- **Простыми словами:** основной язык программирования вашего приложения. JSX — его расширение, где код смешан с разметкой экранов (для React).
- **Суть:** JavaScript — язык, который понимают браузеры; на нём оживает интерфейс. JSX — удобный диалект, где прямо в коде описывают, как выглядит экран. Файлы `.js` и `.jsx` — это JavaScript-часть проекта.
- **Примеры:**
  1. «Логика написана на JavaScript.»
  2. «Экраны — это `.jsx`-файлы.»
  3. «Старый код пока на JS, новый переводим на TypeScript.»

### TypeScript (TS)
- **Простыми словами:** тот же JavaScript, но с проверкой типов данных — он заранее ловит ошибки вроде «сюда должно прийти число, а пришёл текст».
- **Суть:** в обычном JavaScript легко случайно перепутать данные, и ошибка вылезет уже у пользователя. TypeScript добавляет «контролёра», который проверяет это **заранее**, ещё при написании. В вашем проекте принято: новый код — сразу на TypeScript (`.ts`), это надёжнее, особенно там, где деньги.
- **Примеры:**
  1. «Новую утилиту написал на TypeScript.»
  2. «TypeScript поймал ошибку до запуска.»
  3. «`npm run typecheck` проверяет типы.»

### npm
- **Простыми словами:** менеджер, который ставит нужные сторонние библиотеки и запускает команды проекта.
- **Суть:** приложение использует много готовых чужих кусочков (React, иконки и т.д.) — их не пишут заново, а скачивают. npm их устанавливает (`npm install`) и запускает рабочие команды: `npm run dev` (запуск), `npm run build` (сборка), `npm test` (тесты).
- **Примеры:**
  1. «`npm install` ставит зависимости проекта.»
  2. «`npm run dev` запускает локальный сайт.»
  3. «Через npm гоняю тесты перед PR.»

### Линтер (lint, ESLint)
- **Простыми словами:** автоматический «редактор-корректор» кода: проверяет код на ошибки и неаккуратность по общим правилам.
- **Суть:** линтер — как проверка орфографии, но для кода. Он подсказывает: тут переменная не используется, там потенциальная ошибка, здесь не по стандарту. Помогает держать код чистым и единообразным. Запускается `npm run lint`.
- **Примеры:**
  1. «Линтер ругается на неиспользуемую переменную.»
  2. «Перед PR прогоняю линтер.»
  3. «`npm run lint:fix` поправил мелочи сам.»

### Тест (test, Vitest)
- **Простыми словами:** маленькая программа-проверка, которая автоматически убеждается, что код работает правильно.
- **Суть:** вместо того чтобы каждый раз вручную проверять, что расчёт верный, пишут тест — он сам подаёт данные и сверяет результат. Если позже что-то сломается — тест сразу это покажет. Это страховка от случайных поломок. Инструмент тестов у вас — Vitest.
- **Примеры:**
  1. «Написал тест на расчёт состояния статистики.»
  2. «Тесты зелёные — ничего не сломалось.»
  3. «Критичную логику покрываю тестами.»

### Переменные окружения (.env, environment variables)
- **Простыми словами:** секретные настройки (ключи, адреса), которые хранят отдельно от кода и не выкладывают в репозиторий.
- **Суть:** адрес базы и ключ доступа к Supabase — это «ключи от квартиры», их нельзя класть прямо в код (его видят многие). Их держат отдельно — в переменных окружения (`VITE_SUPABASE_URL`, `VITE_SUPABASE_KEY`), а на Vercel задают в панели настроек. Файл `.env` намеренно спрятан от репозитория.
- **Примеры:**
  1. «Ключи Supabase — в переменных окружения, не в коде.»
  2. «Без `.env` приложение не подключится к базе.»
  3. «На Vercel переменные задаются в панели.»

## 6. Общие понятия

### Мок (mock, заглушка, моковые данные)
- **Простыми словами:** временные выдуманные данные «понарошку», пока настоящие из базы ещё не подключены.
- **Суть:** чтобы разрабатывать и показывать экран, не обязательно сразу иметь живую базу — можно подставить «бутафорские» данные (мок). Экран выглядит как настоящий, но цифры зашиты временно. Часть модулей у вас пока на моках (`src/data/`) и постепенно переезжает на реальный Supabase.
- **Примеры:**
  1. «Дашборд пока на моках, не из базы.»
  2. «Моки лежат в `src/data/`, не в компонентах.»
  3. «Заменим мок на реальный запрос к API.»

### MCP
- **Простыми словами:** способ подключить меня к внешним сервисам (Supabase, GitHub, Vercel), чтобы я мог действовать в них напрямую.
- **Суть:** обычно я работаю только с кодом. MCP — это «переходники», через которые я получаю инструменты внешних сервисов: применить миграцию в Supabase, открыть PR на GitHub, посмотреть деплой в Vercel. Без подключённого MCP некоторых действий я просто не могу выполнить.
- **Примеры:**
  1. «Миграцию применю через Supabase MCP.»
  2. «GitHub-действия идут через MCP-инструменты.»
  3. «Если MCP не подключён — нужен коннектор в настройках.»

### Токен (token)
- **Простыми словами:** в зависимости от контекста — либо «электронный пропуск» для доступа, либо «именованное значение цвета/размера» в дизайне.
- **Суть:** слово многозначное. (1) **Токен доступа** — временный цифровой ключ, подтверждающий, что вы вошли (его выдаёт Auth). (2) **Токен дизайна** — именованная настройка оформления (например, «цвет для денег»), чтобы не писать цвет вручную в каждом месте. По контексту обычно понятно, о каком речь.
- **Примеры:**
  1. «После входа выдаётся токен доступа.»
  2. «Зелёный для сумм — отдельный токен темы.»
  3. «Новый цвет добавляю как семантический токен.»

### Баг (bug) и фича (feature)
- **Простыми словами:** **баг** — ошибка, что-то работает не так, как должно. **Фича** — новая возможность/функция приложения.
- **Суть:** два полюса работы. Фича — добавить новое («экран отчётов»). Баг — починить сломанное («сумма считается неверно»). Часто работа делится на «делаем фичи» и «правим баги».
- **Примеры:**
  1. «Нашёл баг: на телефоне кнопка вылезает за край.»
  2. «Новая фича — фильтр по точкам.»
  3. «Сначала закрою баг, потом возьмусь за фичу.»

### Рефакторинг (refactoring)
- **Простыми словами:** наведение порядка в коде без изменения того, что видит пользователь, — чтобы было чище и проще поддерживать.
- **Суть:** со временем код обрастает «костылями». Рефакторинг — это уборка: переписать аккуратнее, разбить громоздкое на части, убрать дублирование. Снаружи ничего не меняется, но дальше работать легче. У вас принцип «оставляй код чище, чем взял» (Boy Scout Rule), без больших разовых переписываний.
- **Примеры:**
  1. «Заодно отрефакторил — вынес повтор в общий компонент.»
  2. «Рефакторинг не меняет поведение, только структуру.»
  3. «Большой рефакторинг не делаем — только по мере касания.»
