### B2. Фонды — 🟡 главное расхождение в данных (не в модели)

- В бою **115 фондов** (49 активных). Кодировка **по точкам и направлениям**:
  `ФД*`(Душанбе), `ФФ*`(Фемали/Fly Garden), `ФМ*`, `ФУ*`(доп. услуги), `ФХ*`(Худжанд?) —
  плюс номер точки в префиксе `1/…`, `2/…`, `3/…`.
- У нас `funds` засеян **эталоном-прототипом ФД1–ФД9/1** (принцип 7 ТЗ — не выдуманные,
  но это *стартовый* набор одной точки).
- 🔴 **Расхождение:** реальные коды фондов (`ФФ7`, `ФМ8/1`, `ФУ3`, `ФХ9`…) у нас не
  заведены. Наш `funds.stage` — **один этап на фонд**; нужно сверить, как этап
  фонда выражен в ManaJet (в API у `FPFund` поля этапа нет — этап живёт в правилах
  распределения; уточнить у поддержки/в `distribution`).
- Балансы фондов в API `FPFund` **не отдаются** → берём из `Register` (свёртка по
  `id_fp_fund`) либо `FpAsset` по периоду.

### B3. Заявки (ЗРС) — ✅ модель совпадает точно

- **ЗРС в ManaJet = `PurchaseOrder`** (не RoutingForm!). Поля:
  `csw_data` / `csw_situation` / `csw_solution` — это в точности наш формат
  **данные → ситуация → решение**, и у нас уже есть `payment_requests.csw_data/
  csw_situation/csw_solution`. ✅
- Привязка к посту (`orgboard_position`), фонду (`fp_fund`), статье (`fp_expense`),
  суммы `planned_value` / `confirmed_value` / `payed_amount`, статус enum `[0,2,3,-1]`.
- Маппинг статусов на наш `request_status`:
  `0` → submitted/planning (одобрено, не оплачено), `2` → approved, `3` → paid,
  `-1` → rejected (по живым данным: -1 = confirmed 0 / paid 0).
- 🟡 RoutingForm/RoutingFormTemplate (общий маршрут согласования) в бою почти пуст
  (1 запись) и у нас отсутствует — для обкатки не нужен.

### B4. Счета — ✅ модель совпадает, 🟡 нет позиций и НДС

| ManaJet | Наша таблица | Статус |
|---------|-------------|:------:|
| Bill (`approvals[]` с `id_fund`/`id_fp_plan`, `planned_date`, `estimated_payment_date`, `total_amount`/`payed_amount`/`remaining_amount`) | `supplier_bills` (два периода: approved/paid) | ✅ совпадает по сути (два периода — наш принцип 4) |
| Invoice (`detail[]` — позиции по `id_stock_card`; `payments[]` — частичные оплаты) | `client_invoices` (банкеты) | 🟡 у нас банкетная модель (зал/гости/событие); у них Invoice **общий** с позициями-номенклатурой |
| 🔴 НДС: `is_vat_enabled`, `vat_calculation_type`, `total_vat_amount`, `vat_id`/`kpp` у контрагента | — нет | 🔴 у нас НДС нигде нет (решить: нужен ли в Таджикистане для наших целей) |
| 🔴 Позиции счёта (`detail[]` → `id_stock_card`, count, price, vat) | — нет | 🔴 счета у нас «одной суммой», без построчной номенклатуры |

### B5. Статистики — 🟡 не хватает «коридора состояний»

- `Stat` в бою: 31 статистика, **28 из 31 имеют коридор `min_val`/`max_val`** и
  `sign` (направление «вверх = хорошо»), `stat_type` (типы: 1=счётная, 8=денежная
  выручка, 11=дивиденд, 12=активы/резервы, + 3,4,9), `period=2` (неделя),
  привязка к посту (`id_position_enters`/`orgboard_position`).
- `StatValue`: значение + `is_quota` (план/квота отдельно от факта), границы периода.
- 🔴 **Наш `statistics` НЕ хранит `min_val`/`max_val` и `stat_type`.** А именно
  коридор задаёт **состояния ХМС** (Власть/Изобилие/Норма/Чрезвычайное/Опасность/
  Несуществование), которые считает `src/utils/stats.ts::calcState`. Сейчас пороги
  состояний нигде в БД не настроены → **это и есть «неправильно настроено».**
  Рекомендация: добавить в `statistics` колонки `min_val`, `max_val`, `stat_type`,
  `unit`, `sign` (у нас есть `invert`), а в `statistic_values` уже есть `is_quota`. ✅

### B6. Оргсхема и люди — 🟡 у нас укрупнённо

## Часть D. План интеграции для обкатки (рекомендация)

> **Обновление по итогу обсуждения:** отдельный раздел «ManaJet · зеркало» убран —
> данные ManaJet встроены прямо в рабочие экраны (Финансы → Заявки, Счета
> поставщиков, Счета клиентам, Фонды; Статистики) через переключатель
> **«Наши данные / ManaJet»** (компонент `src/modules/manajet/MjPanel.jsx`). Сами
> таблицы-зеркало `mj_*` и синхронизация — без изменений.

### D1. Принцип: «зеркало», а не запись в боевые таблицы

Для цели «**смотреть реальные заявки/счета 2–3 недели до запуска**» правильнее
**не лить** данные ManaJet в наши операционные таблицы (`incomes`, `fp_register`,
`payment_requests`, …): там висят триггеры-инварианты (запрет минуса, блокировка
периода, неизменяемость Реестра, защита от двойной оплаты) — массовый импорт 3423
заявок и 2779 счетов с их финансовыми связями будет конфликтовать и исказит наши
балансы.

**Рекомендация:** отдельная схема-зеркало `mj_*` (raw-копии из ManaJet, ключ
синхронизации — `outer_id`/`id` ManaJet), отдельный экран **«ManaJet (зеркало)»** в
приложении. Это:
- безопасно (не трогает наш финконтур и его инварианты);
- быстро ставится и так же быстро убирается после обкатки;
- даёт ровно то, что нужно — **видеть реальные заявки и счета у себя**;
- позже служит источником для «настоящей» миграции в операционные таблицы.

### D2. Что синхронизируем (для обкатки)

Приоритет 1 (то, что просил заказчик): **Bill (счета)**, **PurchaseOrder (заявки/ЗРС)**.
Приоритет 2 (контекст): **FPFund, FpPlan, Stat+StatValue, FpIncome, Invoice,
OrgBoardPosition, Company** (справочники для читаемых названий).

Таблицы-зеркала (пример): `mj_funds`, `mj_periods`, `mj_purchase_orders`,
`mj_bills`, `mj_invoices`, `mj_incomes`, `mj_stats`, `mj_stat_values`,
`mj_positions`, `mj_companies` + одна `mj_sync_log` (когда/что синкали, ошибки).
Каждая: `id` (наш), `mj_id int` (id ManaJet, unique), `data jsonb` (сырой объект),
ключевые поля плоско для фильтров (сумма/дата/статус/фонд), `synced_at`.

### D3. Механика

- **Supabase Edge Function `manajet-sync`** (Deno): ключ ManaJet — в секретах
  Supabase (`MANAJET_AUTH`), на фронт не попадает. Функция ходит на
  `https://api.manajet.org/api/*` (⚠️ **напрямую на manajet.org**, не на
  adminsolution — там 301 теряет авторизацию; см. recon §0), пагинирует
  `filter.skip/take`, делает `upsert` в `mj_*` по `mj_id`.
- **Запуск:** кнопка «Обновить из ManaJet» на экране зеркала (ручной) + опционально
  `pg_cron`/scheduled раз в день. Для 2–3 недель обкатки хватит ручного + ночного.
- **Чтение:** новый раздел в `src/modules/` (карточки заявок/счетов в нашем тёмном
  стиле, адаптив под мобильный — управляющие смотрят с телефона).
- **RLS:** зеркало доступно только финадминам (owner/fin_director) на чтение.

### D4. Объём работ

1. Миграция `mj_*` таблиц + RLS (+ pgTAP по DoD) — через Supabase MCP.
2. Edge Function `manajet-sync` + секрет `MANAJET_AUTH` — деплой через Supabase MCP.
3. API-слой `fetchMj*` в `src/lib/api.js` + экран «ManaJet (зеркало)».
4. PR в `main` → preview-ссылка заказчику.

Альтернатива (если нужно именно «в наших операционных экранах», а не зеркало) —
импорт справочников (фонды/статьи/виды дохода/оргсхема/статистики с коридором) в
операционные таблицы по `outer_id`, **без** транзакционной истории. Это сложнее и
рискованнее; обсуждается отдельно.
