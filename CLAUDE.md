## О проекте

Yakkasaroy Management System — веб-приложение управления сетью ресторанов и туйхон «Яккасарой» (Душанбе · Худжанд · Панджакент) по технологии Hubbard Management System (ХМС): финансовое планирование по фондам, статистики с состояниями, 7-отделенческая оргсхема, CRM банкетов. **Актуальное ТЗ — `docs/ТЗ_Yakkasaroy_Management_System_v2.md`** (v2.0, полная переработка по модели API ManaJet; v1.0 рядом — только история). Обзор модулей и дорожная карта — `README.md`.

**Логика предметной области (первоисточник)** — `docs/manajet-логика.md`: конспект курса «Финансовое планирование» (ATManagement/ХМС), на котором построена ManaJet. Объясняет «почему так»: ФРС и 3 этапа фондов, точка безубыточности (расчёт снизу вверх), еженедельная процедура ФП и роли, система расчётных счетов, ЗРС, ЗП по результату/баллам, дебиторка; в §15 — таблица «концепт курса → реализация в проекте», в §16 — что в финансовом курсе НЕ раскрыто (состояния статистик, ИЦО, шляпа, боевое планирование — это другой слой ХМС).

**Текущая стадия**: на реальном Supabase (через API-слой `src/lib/api.js`, PostgreSQL + RLS, Auth) уже работают модули `finance/*`, `staff`, `org` (оргсхема + шляпы), а также `stats` (ИЦО — `statistics`/`statistic_values`), `dashboard` (`tasks`, `battle_plan_items`, агрегаты), `crm` (`crm_leads`/`crm_clients`/`crm_halls`). На моках в памяти остаётся только `restaurant` (дизайн-референс; по ТЗ v2 §4.11 в продукт не переносится — заменит интеграция iiko). Целевая архитектура по ТЗ v2: Supabase целиком (PostgreSQL + RLS, Auth, Storage, Edge Functions, Realtime), модель данных — по сущностям ManaJet (таблица соответствия в ТЗ v2 §2).

## Рабочий процесс (PR и деплой)

- **Каждое изменение оформляется отдельным Pull Request в `main`** — напрямую в `main` не пушить. Только так изменение получает номер, а Vercel-бот создаёт preview-деплой и присылает заказчику письмо; продакшен на Vercel обновляется после слияния PR в `main`.
- **Один PR за раз: после создания PR — ждать, пока заказчик его смержит, и только потом браться за следующее изменение.** Не плодить очередь открытых PR (заказчик: «не нужно создавать несколько PR, пока не мержну»).
- **В чат заказчику давать ТОЛЬКО короткую preview-ссылку вида `https://yakkasaroy-finance-<hash>-amirtursunov60-hashs-projects.vercel.app/` — это рабочее приложение.** НЕ присылать ссылку на сам PR и НЕ присылать дашборд-ссылку `https://vercel.com/...` (это инспектор деплоя, а не приложение). Короткий адрес берётся из комментария Vercel-бота под PR (поле «Preview») либо из домена готового деплоя. Сам PR всё равно создаётся (без него бот не сгенерирует превью); номером PR можно пометить кратко для трекинга.
- Деплой на Vercel автоматический и привязан к `main`; переменные окружения (`VITE_SUPABASE_URL`, `VITE_SUPABASE_KEY`) задаются в панели Vercel, а не в коде.

## Критерии приёмки (Definition of Done)

Развиваемся эволюционно, без большого рефакторинга: **touch → extract → type → test** (Boy Scout Rule — оставляй код чище, чем взял). Фичи **не останавливаем** ради инфраструктуры — вместо этого подняты критерии приёмки изменений:

- **Новый код — сразу на TypeScript.** Обращения к БД — через типизированный клиент (`supabase` из `lib/supabase.ts`, `createClient<Database>`; типы — `lib/database.types.ts`). Чистую (не-React) логику выносить в модули `.ts` и покрывать unit-тестами (Vitest).
- **Трогаешь старый `.js/.jsx` — попутно переводи затронутое на `.ts`** и дописывай тесты на критичную логику. Не обязательно переписывать весь файл — достаточно того, что меняешь.
- **Изменение схемы БД** (функции/триггеры/RLS/индексы) — сопровождается **pgTAP-тестом инварианта** в `supabase/tests/` (запуск `supabase test db` на ветке/staging) и, при изменении структуры, **перегенерацией `database.types.ts`** (`supabase gen types`).
- **Деньги — только через Реестр `fp_register`.** Инварианты (запрет минуса, блокировка периода, неизменяемость, идемпотентность) держит БД триггерами/констрейнтами/точечными `FOR UPDATE`, а не клиент. Слепой переход на `SERIALIZABLE` не делаем (retry-конфликты) — только точечные блокировки.
- **Перед PR — зелёные** `npm run typecheck`, `npm test`, `npm run lint`, `npm run build`.
- **~20% ёмкости спринта — на операционный контур** (staging, restore-drill бэкапов, плановая сверка/мониторинг), а не только на фичи.

## База данных

**Базовая схема создана в Supabase вне git** (вручную через Dashboard) — полный финконтур по ТЗ v2: 33 таблицы, enum-типы, функции прав, триггеры. Подробное описание и история — `supabase/README.md`. Главное:

- Любое изменение схемы — **только новым файлом миграции** в `supabase/migrations/` (применяются вручную: Dashboard → SQL Editor); применённые файлы не редактировать.
- **Всегда учитывать самую свежую миграцию.** Перед любой работой со схемой свериться с последним по времени файлом в `supabase/migrations/` и со статусами в `supabase/README.md` — строить изменения и код поверх актуального состояния БД, а не baseline. Новую миграцию, ожидающую применения, доводить до применения (через Supabase MCP или Dashboard) и отмечать статус в `supabase/README.md`.
- Источник истины движения денег — Реестр `fp_register`; колонки `balance` фондов и счетов ДС поддерживаются его триггерами (включая запрет минуса и блокировку закрытого периода) — балансы руками не писать, вставка в `incomes` сама создаёт запись реестра.
- Права — RLS на родных функциях `my_role()`, `is_fin_admin()`, `has_location_access()`, `has_fund_access()`, `holds_position()` + таблицы персональных прав (`user_location_access`, `fund_access`, `expense_type_access`). Новые таблицы — только с RLS на этих же функциях.
- Соглашения схемы: `outer_id` (интеграции), `is_archived` (архив вместо удаления), периоды `starts_on`/`ends_on`, ссылки на валюту — `currency_id` (uuid).
- **Supabase MCP: ссылка авторизации (OAuth) одноразовая и быстро протухает** — перед применением миграции вызывать `authenticate` заново и давать заказчику СВЕЖУЮ ссылку (старая даёт `Unrecognized client_id`). Не переиспользовать ссылку из прошлого сообщения/сессии.
- **Миграции применяет Claude через Supabase MCP — заказчик SQL вручную не запускает** (его явное требование). Для этого Supabase MCP-сервер (`.mcp.json`, `mcp.supabase.com`) должен быть подключён к сессии. В облачной сессии Claude Code на вебе серверы из `.mcp.json` сами не поднимаются — их нужно включить как коннектор в настройках окружения (Environment → MCP/Connectors), иначе инструментов `authenticate`/`apply_migration`/`execute_sql` в наборе нет и ссылку авторизации сгенерировать нечем. Если MCP не подключён — не предлагать ручной Dashboard как основной путь, а попросить включить коннектор.

## Предметная область (принципы из ТЗ v2 — менять нельзя)

1. Распоряжаемся только фактически имеющимися средствами. Операция дохода сразу указывает, куда физически пришли деньги (счёт ДС + способ оплаты); Контроль средств — производная от этих привязок, а не параллельный учёт.
2. Все деньги при поступлении распределяются по фондам по схемам в 3 этапа: от выручки → от маржинального дохода → от скорректированного дохода. **Этап — свойство правила распределения.** _Уточнение заказчика (`docs/funds-spec.md`): этап задаётся как одно свойство фонда (`funds.stage`), «домашний этап» в Директиве = дефолтное правило `distribution_rules`; смена этапа фонда синхронизируется RPC `fp_set_fund_stage`. Прежнее правило «ФД6 пополняется с двух этапов сразу» отменено — фонд одноэтапный._
3. Расход — только из фонда через одобренную заявку или счёт. Заявка подаётся **от поста оргсхемы** (не «от пользователя») и обязательно в формате **ЗРС**: данные → ситуация → предлагаемое решение.
4. ФП проводится еженедельными периодами (чт–ср), период закрывается Директивой; операции в закрытом периоде запрещены. Счёт поставщика живёт в **двух периодах**: одобрения и оплаты (могут отличаться).
5. **Реестр — единая лента всех операций ФП** и источник истины: балансы фондов и счетов ДС — производные леджера, запрет ухода в минус на уровне БД.
6. Каждая точка учитывается отдельно, владелец видит консолидированную картину. Точка/филиал — первоклассная сущность, любая запись привязывается к точке.
7. Справочники прототипа (фонды ФД1–ФД9/1, статьи РД1–РД9, D-коды видов дохода, 7 отделений, состояния и коэффициенты ЗП) — реальные стартовые данные системы, не выдуманные примеры: не переименовывать и не «чистить».

## Правила для кода

- **Язык интерфейса — русский.** Все надписи, метки, комментарии в коде — на русском. Архитектурно заложен i18n (`lang`: ru/tj), но строки пока в коде.
- **ХМС-термины не переводить и не переименовывать** — ни в UI, ни в именах данных: ФП, ФРС, фонд, Директива, ЦКП, ИЦО, ЗРС, шляпа, статистика, квота, Реестр, состояния (Власть, Изобилие, Норма, Чрезвычайное положение, Опасность, Несуществование), боевое планирование, туйхона, точка. Глоссарий — Приложение А в ТЗ v2.
- **Моки лежат в `src/data/`** — новые демо-данные добавлять только туда, не зашивать в компоненты. Этот слой целиком заменится API при подключении Supabase, поэтому компоненты не должны мутировать данные напрямую сверх локального state.
- **Стиль — тёмная тема в духе Alif Business** (зелёные акценты, цвет по умолчанию dark). Все цвета берутся только из палитры `C` (`THEMES` в `src/theme/theme.js`) — обе темы (dark/light) должны работать; никаких захардкоженных цветов в компонентах.
- **Инлайн-стили из `src/theme/`** — основной способ оформления ядра: общие стили добавлять в `makeStyles` (`styles.js`) и использовать как `st.имя`; hover/анимации — в `makeCss` (`css.js`) через классы. Существующие экраны на инлайн-стилях не переписывать ради миграции — только по мере касания (Boy Scout).
- **Tailwind v4 + shadcn — разрешены для НОВЫХ компонентов/экранов** (решение владельца, фундамент — PR с Tailwind). Не «или-или», а сосуществование:
  - Tailwind подключён в `src/index.css` **БЕЗ preflight** (слои `theme` + `utilities`), чтобы глобальный reset не ломал инлайн-стили текущих экранов. Поддеревья на Tailwind/shadcn оборачивать в `className="tw-scope"` (там включён border-box и дефолт рамки).
  - **Цвета — единый источник: палитра `C`.** Tailwind-классы (`bg-panel`, `text-money`, `border-line`, `text-brand/info/warning/success/danger`…) ссылаются на CSS-переменные `--c-*`, которые `App.jsx` выставляет из активной темы (`applyThemeVars` в `theme.js`). Хардкод цветов в классах не вводить — пользоваться семантическими токенами из `@theme` (`src/index.css`). Новые семантические цвета добавлять в три места синхронно: `THEMES` (`theme.js`), `TW_VAR_KEYS` (`theme.js`), `@theme` (`index.css`).
  - shadcn: `components.json` настроен, алиас `@/` (=`src/`), `cn` — в `src/lib/utils.ts`. Готовые компоненты: `npx shadcn@latest add <component>` (появятся в `src/components/ui`), затем адаптировать под нашу тёмную тему через мостовые токены.
  - Мобильная адаптация и тёмная тема обязательны и для Tailwind-компонентов (правила ниже без изменений).
- CSS-файлы заводить можно (рядом с компонентом, как `switcher.css`); глобальные — через `src/index.css`.
- Графики — собственные SVG-компоненты в `src/components/charts/`, библиотеки графиков не подключать.
- **Мобильная адаптация — обязательная часть КАЖДОГО изменения, а не отдельная задача.** Управляющие работают с телефона, поэтому любой новый/правленый экран, таблица, форма, шапка или модал сразу делается адаптивным под `isMobile` (из `useIsMobile`) — без отдельной команды заказчика. Правила по умолчанию:
  - Широкие таблицы: на телефоне ключевые колонки (название/сумма/действие) помещаются в ширину экрана, второстепенные — за ними с горизонтальным скроллом; на телефоне сетка `gridTemplateColumns` своя (уже фиксированных колонок, меньше паддинги).
  - **Списки сущностей (фонды, счета и т.п.) — предпочитать карточки, а не горизонтально-скроллящиеся таблицы** (рекомендация заказчика, образец — вкладка «Фонды»): каждая сущность = адаптивная карточка (сетка `repeat(auto-fill, minmax(~300px, 1fr))`, на телефоне в одну колонку), все ключевые цифры и действия видны без горизонтального скролла; действия — кнопками прямо на карточке.
  - Кнопки/тулбары в шапках переносятся (`flexWrap`), длинные подписи на телефоне сокращаются или прячутся за иконку.
  - Модалы — ширина `min(Xpx, 100%)`; формы на телефоне в одну колонку.
  - Ничего не должно вылезать за правый край и обрезаться (проверять правый блок шапки, аватар, чипы).
  - После любой UI-правки мысленно (или прогоном) проверять обе ветки — десктоп и `isMobile` — и доводить мобильный вид сразу, не дожидаясь замечаний «отодвинь/сократи».
- Денежные суммы форматировать утилитами из `src/utils/format.ts`: **`fmt`** — точная сумма до дирамов (Реестр, Директива, формы), **`fmtShort`** — сокращённо (млн/тыс) для дашбордов и обзорных карточек. Основная валюта — TJS (сомони).
