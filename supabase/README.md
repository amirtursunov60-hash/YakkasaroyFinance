# База данных (Supabase)

Вся схема БД живёт в `migrations/` — пронумерованные SQL-файлы. Правило: **схему меняем только новым файлом миграции** (005, 006, …), уже применённые файлы не редактируем — иначе база и git разойдутся.

## Как применить (через Dashboard, без установки чего-либо)

1. Откройте [supabase.com/dashboard](https://supabase.com/dashboard) → ваш проект → **SQL Editor**.
2. Скопируйте содержимое файла `migrations/001_auth_profiles.sql`, вставьте в редактор, нажмите **Run**.
3. Повторите по порядку для `002_core.sql`, `003_finance.sql`, `004_seed.sql`.
4. Проверка: в **Table Editor** должны появиться таблицы `locations`, `funds`, `income_types`, `incomes`, `fp_register` и др., а в `funds` — 12 фондов ФД1–ФД9/1.

Все файлы идемпотентны — повторный запуск ничего не сломает. `001` безопасен для уже существующей вручную таблицы `profiles`.

## Что в миграциях

| Файл | Содержимое |
|---|---|
| `001_auth_profiles.sql` | `profiles` (фиксация ручной настройки), роли, `app_role()`, RLS |
| `002_core.sql` | точки (`locations`), валюты, способы оплаты, периоды ФП чт–ср (`fp_periods` + `ensure_fp_period(date)`), счета ДС (`cash_accounts`) |
| `003_finance.sql` | фонды, виды дохода (D) и расхода (РД), правила распределения (этап — свойство правила), операции дохода (`incomes`), Реестр (`fp_register`), триггеры: доход→реестр, запрет операций в закрытом периоде |
| `004_seed.sql` | стартовые данные из прототипа: 6 точек, 12 фондов, схема распределения 3 этапов, дерево D-кодов, статьи РД, кассы, валюты |

## Ключевые принципы схемы (из ТЗ v2 §2.3)

- **Реестр (`fp_register`) — источник истины**: балансы фондов и счетов ДС считаются суммой по реестру, отдельно не хранятся.
- Операция дохода обязана указывать **счёт ДС и способ оплаты** — запись в реестре создаётся триггером автоматически.
- Этап распределения — в `distribution_rules`, не в фонде (ФД6 пополняется с этапов «выручка» и «маржинальный» одновременно).
- Операции в закрытом периоде блокируются на уровне БД.
- Везде `outer_id` (интеграции iiko/импорт) и `in_archive` (архив вместо удаления).

## Доступ из фронтенда

Клиент уже настроен (`src/lib/supabase.js`), нужны `VITE_SUPABASE_URL` и `VITE_SUPABASE_KEY` в `.env`. Права обеспечивает RLS: чтение справочников — всем вошедшим, закрытые фонды (ФД5/ФД6/ФД7) — только `owner`/`fin_director`, ввод доходов — `owner`/`fin_director`/`location_manager`/`accountant`.
